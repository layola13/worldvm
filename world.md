## 第一章：设计哲学与架构宏图

### 1.1 深度学习的困境与符号主义的复兴
现代基于 Transformer 的大型语言模型 (LLM) 采用“全量激活”的稠密矩阵乘法。这种架构本质上是将“知识”硬编码进“逻辑”权重中，导致了以下灾难性问题：
*   **效率极低**：执行简单的逻辑推理（如 A > B）需要调动千亿参数。
*   **算力垄断**：极度依赖 GPU 的海量显存带宽 (HBM) 与并发 ALU。
*   **幻觉频发**：依靠概率分布而非严密的因果链，缺乏底层物理世界的约束。

WVM 旨在复兴符号逻辑 (Symbolic AI)，并结合现代 ECS (Entity Component System) 与 DOD (Data-Oriented Design) 架构，彻底抛弃浮点矩阵。

### 1.2 核心理念：推理即物理拓扑的离散流转
在 WVM 中：
*   **概念即实体**：所有的具象物体（如苹果）和抽象概念（如重力、公平）都被编码为定长的 16x16x16 二进制位图 (Voxel Bitmask)。
*   **推理即碰撞**：推演不再是计算向量距离，而是让实体在 32³ 的局部沙盒中执行按位与 (`Bitwise AND`) 的物理碰撞。
*   **逻辑即物理**：逻辑矛盾表现为体素穿模；逻辑连贯表现为体素的完美咬合。

### 1.3 100MB 极限约束下的系统大一统
系统在 100MB 的内存约束下，统一了宏观物理计算与微观心理推演。同样的 C/Rust 位运算底层代码，既能处理“石头砸碎玻璃”的物理现象，也能处理“一句话击碎世界观”的心智崩溃。

## 第二章：内存布局与极限预算 (The 100MB Ledger)

系统采用极限压榨的静态内存分配策略，拒绝一切带有垃圾回收 (GC) 的运行时。

### 2.1 内存空间全景分布表

| 模块名称 | 描述与用途 | 预算大小 | 占比 |
| :--- | :--- | :--- | :--- |
| **Engine Runtime** | 物理光线投射、ECS 调度器、AST 解析器前端 (机器码) | ~10 MB | 10% |
| **ROM: L1 字典** | 标准世界观基元，存储 2500 个 4KB 大小的 `Entity_16³` | 10 MB | 10% |
| **ROM: 知识与行为树** | 词汇哈希表、DAG 计划模板、肢体微指令、情绪触发表 | 10 MB | 10% |
| **RAM: 动态沙盒池** | 并发维持 1000 个 32³ 事件沙盒 (物理沙盒 + 影子沙盒) | ~5 MB | 5% |
| **RAM: 关系记忆图谱** | 存储上百万条节点连线（短期与长期语义网络） | 15 MB | 15% |
| **Reserved 战略储备** | 预留空间，可用于技能包(Cartridges)动态加载或历史快照缓存 | 50 MB | 50% |

### 2.2 L1 缓存命中率与零开销抽象
实体的 16³ 位图严格限制为 512 Bytes (即 64 个 `u64`)。这完美对齐现代 CPU 的 Cache Line (通常为 64 Bytes)。读取一个实体的全景拓扑，CPU 只需要发出 8 次完美的连续内存预取，实现真正的零开销地址跳转。

## 第三章：核心数据结构 (Data-Oriented Design)

宏观实体与心理概念共享完全相同的 `Entity_16x16x16` 结构体，固定占用 4096 Bytes。

### 3.1 `Entity_16x16x16`：4KB 基元的位组装
```c
#pragma pack(push, 1) // 强制 1 字节对齐
struct Entity_16x16x16 {
    // [区块 A: 拓扑形态] - 512 Bytes
    u64 local_topology[64]; 
    
    // [区块 B: 物理法则] - 8 Bytes
    u16 mass;              // 质量
    u16 hardness;          // 硬度
    u8  material_type;     // 材质掩码 (刚体/流体/气体)
    u8  collision_group;   // 碰撞分组
    u16 flags;             // 静态/活动/可燃等 bool 状态

    // [区块 C: 逻辑调色板] - 1 Byte
    // 高 4-bit 主色，低 4-bit 辅色 (0:透明, 1:红, 2:绿... 9:紫)
    u8  color_pack;        

    // [区块 D: 心理 4 维寄存器] - 4 Bytes
    i8  valence;           // 正负能量 (-128 to 127)
    u8  arousal;           // 激活度 (0 to 255)
    u8  certainty;         // 确定性 (0 to 255)
    u8  control;           // 可控感 (0 to 255)

    // [区块 E: 关系连线图] - 192 Bytes
    u16 relational_edges[64]; // 指向其余实体的 ID (0x0000 - 0xFFFF)
    u8  edge_weights[64];     // 连线强度或衰减率

    // [区块 F: 算子触发器] - 8 Bytes
    void (*on_collide_rules)(void*, u16, u16); 
    
    // [区块 G: 保留填充] - 约 3371 Bytes
    u8 padding[3371];
};
#pragma pack(pop)
```

### 3.2 逻辑调色板混色机制
采用 10x10 LUT 二维数组，避免浮点色彩插值：
```c
const u8 COLOR_MIX_LUT[10][10] = {
    // 预计算的混色矩阵，如 蓝(7)+黄(5)=绿(6)
};
```
引擎仅用 1 个 CPU 周期即可完成光学推演。

## 第四章：事件视界与时空调度 (The 32³ Sandbox)

系统不设全局坐标，一切计算都在动态开辟的局部沙盒中进行。

### 4.1 局域拓扑沙盒的动态生成
当需要处理动词（如“锤子砸玻璃”）时：
1.  **分配空间**：从预分配的 5MB 沙盒池中弹出一个空闲的 `Scene_32x32x32` 结构体。
2.  **基准锚定**：将被动方（玻璃）置于沙盒中央 `(X:8, Y:8, Z:8)`。
3.  **切入计算**：主动方（锤子）根据算子指令附加的相对侵入向量，实例化在沙盒边缘并带有初始动量意图。

### 4.2 Instance 实例指针池
沙盒内部的 `instances` 数组仅存储指针和动态变换，避免 16³ 拓扑的重复拷贝：
```c
struct Instance {
    u16 entity_id;      // 指向 L1 ROM 字典中的原型
    i8  pos_x, pos_y, pos_z; // 在 32³ 中的局部坐标
    u8  rot_yaw, rot_pitch, rot_roll; // 罗盘旋转角度
    u8  sleep_tick;     // LOD 休眠计数器
};
```

## 第五章：四步心跳引擎 (The 4-Step Tick Pipeline)

沙盒推演由严格的离散逻辑帧 (Tick) 驱动。

### 5.1 第一步：意图收集 (Gather)
扫描沙盒内所有实例。结合全局变量（重力向下，风力等）与当前速度，计算出本帧的 `Intent_Vector`（如 `dy = -1`）。

### 5.2 第二步：位移试探 (Speculative Shift)
构建临时的 32³ `test_mask`。将实例的 16³ 原型位图根据当前的 `pos` 与 `Intent`，通过按位移位 (`>>` 或 `<<`) 映射到 32³ 位图上。
```c
// 伪代码：极其廉价的位运算替代多边形碰撞
u64 collision_result[512];
for(int i=0; i<512; i++) {
    collision_result[i] = scene->world_mask[i] & test_mask[i];
}
```

### 5.3 第三步：碰撞决断 (Resolution)
如果 `collision_result` 为全 0，接受意图，更新实体局部坐标。
如果非 0，判定为碰撞发生。提取双方 `hardness` 属性；硬度低的一方触发其原型的 `on_collide_rules` 函数，可能导致状态改变（如颜色变暗，位图破碎分解为多个新实例）。

### 5.4 快进机制 (Fast-Forward)
若场景内所有实体的意图计算不依赖复杂的 I/O 中断，CPU 可以用纳秒级速度在一个 `while(unstable)` 循环中空转上千个 Tick，直到所有物体动能为 0，直接向外侧输出最终态。

## 第六章：时空双重降维机制 (Spatio-Temporal LOD)

此机制是维持低端 CPU 也能实现庞大世界推演的核心。

### 6.1 视觉聚焦模糊 (Foveated Blurring)
由算子指令（如 `OP_SMASH`）决定焦点。
*   **Fovea 核心区**：参与完整的 `u64[64]` 位图展开与按位碰撞。
*   **Periphery 边缘区**：只提取 `Instance` 结构体中的 `pos_x, y, z` 以及简化的 AABB 半径。碰撞退化为粗略的欧式距离比较。

### 6.2 异步休眠与唤醒 (Sleep & Wake)
*   引擎赋予焦点实例 `sleep_tick = 0`（每帧更新）。
*   赋予边缘实例 `sleep_tick = 10`（每 10 帧更新一次）。
*   **唤醒**：当焦点区产生的破片（高动能实例）飞越网格边界并进入边缘实例的 AABB 包围盒预警范围时，触发硬件中断。边缘实例被立刻“解包 (Unpack)”至焦点状态，恢复高频心跳。

## 第七章：指令集与算子路由 (The Bagua ISA)

自然语言被编译为 6-bit 的底层指令（六十四卦算子）。

### 7.1 六十四卦算子映射
利用 C 语言的函数指针数组 (Jump Table) 实现 O(1) 路由。
```c
void (*BAGUA_ISA[64])(Scene_32x32x32* scene, u16 actor_id, u16 target_id) = {
    [0x00] = op_idle,
    [0x1A] = op_fuse,    // 泰卦：融合/化合
    [0x3B] = op_smash,   // 夬卦：破坏/决裂
    [0x0F] = op_avoid,   // 遁卦：规避/远离
    [0x3C] = op_repair,  // 节卦：连接/修复
    // ...
};
```
执行语句 `BAGUA_ISA[opcode](sandbox, id1, id2);` 即可驱动沙盒运算。

## 第八章：心智引擎与情绪流转 (The Mental Engine)

心理推理与物理碰撞在底层架构上完全统一。

### 8.1 思考即影子沙盒空转
“思考”操作会从主沙盒 `fork()` 一个 32³ 影子沙盒进入私有内存区，在其中进行高频 `Fast-Forward` 的动作模拟。模拟结果通过连线图传回主世界，引导决策（如避开危险）。

### 8.2 情绪 4 维寄存器
每个实体携带 `valence`(正负), `arousal`(激活), `certainty`(确定), `control`(控制感) 寄存器。

### 8.3 Fear 与 Love 的底层调度表现
*   **Fear (恐惧)**：`arousal` 极高。触发“隧道视觉”。
    *   *拓扑*：紧缩核带尖刺，碰撞强制返回 `AVOID/FREEZE`。
    *   *LOD*：强行劫持引擎调度，焦点半径降至最低，焦点区 Tick 频率飙升（集中算力），周围区域完全冻结（忽略复杂关联推理）。
*   **Love (爱/安全)**：`valence` 与 `control` 高。
    *   *拓扑*：同心环稳定核。碰撞倾向于触发 `FUSE/BIND`。
    *   *LOD*：焦点温和扩大，允许跨越更远距离的实体位图在沙盒中进行掩码交叠，促进概念创新与联想（触类旁通）。

## 第九章：肢体语言与行动计划 (Kinesics & AST DAG)

将沟通与长线任务降维为微字节码脚本，拒绝占用存储空间的 3D 动画。

### 9.1 肢体动作微指令序列 (Microcode)
“挥手”、“握拳”等肢体语言是应用于 16³ 骨架节点的一组时空变换差值（Deltas）：
```c
struct Action_Microcode {
    u16 target_entity;   // 操作部位 (如右手)
    u8  transform_type;  // PITCH, YAW, ROLL, TRANSLATE
    i8  magnitude;       // 步进幅度
    u8  tick_duration;   // 动作帧数
    u8  next_action_idx; // 链式索引
};
```
肢体的拓扑紧缩（如握拳）会直接被碰撞系统识别，拉高局部 `arousal` 寄存器，硬件绑定触发“愤怒/警戒”心理状态。

### 9.2 多 Lane 确定性并行调度器 (DAG Scheduler)
计划（如“做饭”）存储为有向无环图 (DAG)。
*   引擎内部维持独立执行通道：`Left_Hand_Lane`, `Right_Hand_Lane`, `Speech_Lane`。
*   每个 Tick，调度器检查 DAG 前置依赖。独立节点可在不同的 Lane 中并行压入沙盒，避免单线程堵塞（实现“左手拿刀，右手拿面包”的并行）。

## 第十章：编译器前端与系统拔靴 (Bootstrapping)

如何构建起最初的 2500 个 16³ 核心基元字典？

### 10.1 降维 Prompt 与大模型赋能
采用现有 LLM (如 Claude/Gemma) 作为“算力苦力”。
1.  **定义基元集**：设计 64 种基础空间拓扑（线、面、壳、核）。
2.  **批量注入 Prompt**：
    `"请将【椅子】拆解为核心物理组件和相对坐标。不解释，只输出组件数组。"`
3.  **二次编译**：写一个 Python 脚本，将 LLM 输出的结构化 JSON/数组直接编译为二进制的 512B `u64[64]` 位图数据，写入 `std::world` 二进制 ROM 文件。

### 10.2 NLP 到 AST 的映射
系统前端维护一个轻量级的词典 Hash 表。接收自然语言后，匹配并查出对应实体 ID 与 6-bit 算子指令，打包为 `Event_Command` 结构体，推入引擎队列。从此，宏观世界与心理世界的时空齿轮开始转动。
