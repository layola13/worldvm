# 21 3D 引擎总览与执行模型

## 1. 章节定位
本章专门回答你在 `chat.md` 里反复强调的核心：  
WVM 的“3D 引擎”不是传统 GPU 图形管线，而是**CPU 事件驱动的体素推演引擎**。

## 2. 3D 引擎核心定义
WVM 3D 引擎由四个不可拆分部件组成：
1. `Entity16`：16³ 体素原型（ROM 中只读定义）。
2. `Scene32`：32³ 局部事件沙盒（RAM 中临时运行态）。
3. `Spatial Scheduler`：空间调度器（实例注入、焦点路由、LOD 升降级）。
4. `Tick Engine`：四步心跳执行器（Gather/Speculate/Resolve/Commit）。

## 3. 与传统 3D 引擎的边界
### 3.1 本引擎保留的能力
1. 空间表示（体素占位）。
2. 变换（平移、旋转、缩放映射）。
3. 碰撞（AABB + 位图掩码）。
4. 物理算子（重力、流体、材料状态）。
5. 视图输出（ASCII/结构化观测）。

### 3.2 本引擎刻意舍弃的能力
1. 复杂材质系统（PBR）。
2. GPU Shader 链路。
3. 高分辨率纹理与后处理特效。

目的很明确：把算力预算留给“可解释推理”而不是“像素特效”。

## 4. 16³ in 32³ 的空间层级
你在对话里提出的关键比例已纳入标准：
1. 单体定义：`16x16x16`（4096 bit，512B）。
2. 事件空间：`32x32x32`（32768 bit，4KB）。
3. 运行机制：在 32³ 场景中注入多个 16³ 实例，通过变换与掩码运算完成交互。

这一比例的价值：
1. 计算稳定：单体规模固定，便于指令优化。
2. 调度高效：场景以页级块管理，方便对象池复用。
3. 扩展简单：多 Scene 并发时成本可预测。

## 5. 事件驱动而非全局常驻
WVM 3D 引擎遵守“局部事件视界”原则：
1. 只有输入事件到来时创建 Scene32。
2. 只装载当前事件相关实体。
3. 达到稳定态即销毁场景并提交 Delta。

这直接解决了传统仿真引擎“后台空转占用高”的问题。

## 6. 指针引用 + 零拷贝实例化
3D 引擎实例化遵循“ROM 原型只读，RAM 持指针”：
1. Scene 只保存 `entity_id + transform + state`。
2. 原型拓扑始终留在字典，不复制大块数据。
3. 旋转采用 LUT 索引，避免每 Tick 重算三角函数。

## 7. 空间调度器职责
空间调度器（Spatial Scheduler）不做渲染美化，只做三件事：
1. 把实体按优先级注入焦点区/外围区。
2. 决定哪些对象需要全精度碰撞，哪些对象近似处理。
3. 输出本 Tick 需要执行的空间操作列表。

## 8. 引擎执行主循环
```c
for each event:
    scene = alloc_scene32()
    inject_instances(scene, event)
    while !stable(scene):
        run_tick(scene)
    delta = build_delta(scene)
    commit_delta(delta)
    release_scene(scene)
```

## 9. 3D 引擎与语义引擎耦合点
1. Parser 只负责编译“动作意图”，不直接做空间运算。
2. 空间引擎负责把动作意图落成真实几何变化。
3. 心智层通过影子沙盒调用同一 3D 执行器做预演。

## 10. 必须实现的 3D 核心算子（首版）
1. `SPAWN_INSTANCE`
2. `DESPAWN_INSTANCE`
3. `MOVE_DELTA`
4. `ROTATE_LUT`
5. `TEST_COLLISION`
6. `APPLY_FORCE_FALL`
7. `APPLY_FLOW`
8. `COMMIT_GEOMETRY_DELTA`

## 11. 性能目标（3D 引擎维度）
1. 单 Scene 初始化 < 1ms
2. 单 Tick（轻场景）< 100us
3. 空闲状态接近零占用
4. 多 Scene 并发不出现长尾卡顿

## 12. 本章结论
WVM 的 3D 引擎本质是“体素空间调度 + 离散物理算子 + 事件驱动生命周期”。
