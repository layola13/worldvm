# 15 安全、对齐与可解释性

## 1. 目标
确保 WVM 在执行层面“可控、可审计、可回滚”，避免不可预测行为扩散。

## 2. 安全层位置
安全层插在三个阶段：
1. IR 生成后（静态审查）。
2. 指令执行前（运行时守卫）。
3. 提交前（结果审计）。

## 3. 策略守卫（Policy Guard）
每条指令执行前必须检查：
1. 对象合法性（actor/target 是否存在）。
2. 权限合法性（是否允许操作）。
3. 风险预算（是否超资源阈值）。
4. 禁止规则（黑名单动作组合）。

## 4. 对齐策略
1. 对齐规则优先级高于计划节点优先级。
2. 对齐规则不可被情绪寄存器覆盖。
3. 不确定场景必须触发澄清而非猜测执行。

## 5. 解释链标准
解释链应包括：
1. 输入摘要
2. 解析结果
3. 计划 DAG
4. 实际执行指令序列
5. 关键状态变化
6. 最终结果与原因码

## 6. 日志分级
1. `TRACE`：Tick 级细节（调试使用）。
2. `INFO`：关键状态转移。
3. `WARN`：回滚、重规划、冲突。
4. `ERROR`：执行失败、策略拒绝。

## 7. 回滚与补偿
1. 对可逆动作优先回滚。
2. 对不可逆动作执行补偿计划。
3. 回滚失败必须进入 fail-safe 状态。

## 8. Fail-safe 模式
触发条件：
1. 连续异常超阈值。
2. 安全守卫多次拦截关键流程。
3. 关键数据损坏。

行为：
1. 停止非必要执行。
2. 输出保守响应。
3. 请求外部确认。

## 9. 风险分类
1. 逻辑风险：计划冲突、死锁、震荡。
2. 安全风险：越权操作、敏感行为误触发。
3. 认知风险：低确定性下强执行。
4. 数据风险：记忆污染、版本不一致。

## 10. 风险缓解
1. 规则白名单优先。
2. 高风险动作双重确认。
3. 关键路径强制记录解释链。
4. 定期进行异常回放审计。

## 11. 指标
1. 策略拦截率
2. 误拦截率
3. 回滚成功率
4. 可解释覆盖率（有解释链的执行占比）
5. 高风险动作确认率

## 12. 审计接口建议
提供查询接口：
1. 按 `event_id` 拉取完整执行链。
2. 按时间窗口拉取异常事件。
3. 按规则 ID 查询触发频次。

## 13. 本章结论
WVM 的优势不只是省算力，更是“可治理”。  
安全、对齐、可解释性必须是执行路径的一等公民。
