# 08 训练部署与工程里程碑

## 1. 目标
整理 `todo2.md` 的训练与部署讨论，给出从内核到模型协作的工程节奏。

## 2. 三层系统
1. **执行层**：Zig 100MB 逻辑内核（真值裁判）。
2. **翻译层**：7B~9B 逻辑编译模型（语义 -> 算子）。
3. **生成层**：大模型主对话系统（用户交互）。

## 3. 训练目标
翻译层模型重点学习：
1. 从日常语言稳定映射到 `kernel_op`。
2. 识别典型物理违规并给出纠错解释。
3. 在错误反馈下进行二次改写。

## 4. 数据阶段划分
1. P1（冷启动）：1k~5k 高质量样本验证格式与闭环。
2. P2（扩张）：10k~50k 覆盖更多场景和错误类型。
3. P3（量产）：100k 级别构建稳定纠错能力。

## 5. 训练策略（抽象级）
1. 优先 LoRA/QLoRA 快速迭代。
2. 关键版本可做更高秩或全参数训练（资源允许时）。
3. 始终保留“正确+错误”双样本结构。

## 6. 推理部署建议
1. 首版部署：本地/服务端模型 + Zig 动态库协同。
2. 模型输出先过 `kernel_op` 校验，再走物理审计。
3. 审计失败返回可解释错误，而非静默失败。

## 7. 运行模式
1. **开发模式**：完整 trace + ASCII 回放。
2. **生产模式**：摘要审计 + 关键失败留档。
3. **训练采样模式**：随机抽取失败例回流数据工厂。

## 8. 工程里程碑
1. M1：1024³ 分页 + 基础碰撞 + CLI 演示。
2. M2：Tool Call 接入 + 样本自动生成器。
3. M3：翻译层模型首训 + 纠错闭环上线。
4. M4：混合 Hook + 规模化数据飞轮。

## 9. 验收指标
1. 物理审计拦截准确率。
2. 错误纠正成功率。
3. 推理延迟与吞吐。
4. 训练数据有效率（可执行比例）。
5. 回归场景稳定率。

## 10. 风险与缓解
1. 风险：数据量大但有效率低。  
缓解：先小规模严筛再扩张。
2. 风险：模型学会“格式”但不会“物理”。  
缓解：提高错误样本质量与审计反馈密度。
3. 风险：Hook 侵入过强损伤自然表达。  
缓解：分级拦截，仅拦高风险语义。

## 11. 结论
`todo2.md` 路线的本质是：  
用 Zig 内核建立真值边界，用数据飞轮逐步把模型拉回物理一致性轨道。
