# 13 沟通、手语与肢体语言

## 1. 目标
构建“日常沟通优先”的知识与动作体系。  
从语言、手语、肢体动作三路输入统一映射到同一执行语义。

## 2. 统一原则
1. 沟通最终表达为可执行语义指令，而非文本概率续写。
2. 手语与肢体不是视频帧，而是微指令序列。
3. 同一意图可有多模态入口，共享一个 IR。

## 3. 输入通道
1. 文本通道：自然语言句子。
2. 手语通道：离散手势 token 流。
3. 肢体通道：动作模式 token（点头、摇头、指向）。

## 4. 动作微码模型
每个动作定义为短序列：
```c
typedef struct {
    uint16_t actor_part;   // 头/左手/右手/躯干
    uint8_t  transform;    // 旋转/平移/展开/收缩
    int8_t   magnitude;
    uint8_t  duration;
    uint8_t  next_idx;
} MicroAction;
```

## 5. 推荐基础动作集（首版）
1. `NOD`（点头）
2. `SHAKE_HEAD`（摇头）
3. `WAVE`（挥手）
4. `POINT`（指向）
5. `OPEN_PALM`（摊掌）
6. `CLENCH_FIST`（握拳）
7. `STEP_BACK`（后退）
8. `STEP_FORWARD`（前进）

## 6. 语义映射示例
1. 点头 -> `CONFIRM`
2. 摇头 -> `DENY`
3. 指向 + 视线聚焦 -> `SELECT(target)`
4. 摊掌 -> 低威胁表达，提升对话可达性
5. 握拳 -> 警戒/防御倾向，可能提高 arousal

## 7. 语用上下文
同一动作在不同上下文意义不同，需绑定上下文槽位：
1. 场景角色（说话者/听话者）。
2. 目标对象。
3. 当前计划状态。
4. 情绪状态。

## 8. 沟通回路
1. 输入解析为语义 IR。
2. 执行层更新世界/心智状态。
3. 输出层生成语言或动作反馈。
4. 若确定性不足，触发澄清动作 `ASK`。

## 9. 常见场景模板
### 9.1 日常问候
`FOCUS(person) -> WAVE -> ASK(status) -> CONFIRM`

### 9.2 请求物品
`POINT(item) -> ASK(request) -> WAIT -> CONFIRM/RETRY`

### 9.3 拒绝与回避
`SHAKE_HEAD -> DENY -> MOVE_AWAY(optional)`

## 10. 手语实现建议
1. 定义手势词表（token -> 语义槽）。
2. 用时间窗口识别手势序列。
3. 将序列映射到 microcode 或直接映射到 opcode。
4. 保留歧义候选，交给上下文决策。

## 11. 表达策略
输出优先级建议：
1. 先给结论动作（点头/摇头/指向）。
2. 再给简短语言解释。
3. 复杂情况给结构化澄清问题。

## 12. 误解处理
1. 检测冲突信号（动作与语言不一致）。
2. 触发 `ASK_CLARIFY`。
3. 更新关系权重与确定性寄存器。

## 13. 指标
1. 意图识别准确率
2. 手势歧义率
3. 澄清触发率
4. 沟通成功率
5. 端到端时延

## 14. 本章结论
沟通系统要先“可执行”再“可修饰”。  
手语与肢体语言是降低歧义、提升鲁棒性的关键输入通道。
