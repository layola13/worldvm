# 07 四步 Tick 流水线

## 1. 目标
定义 WVM 的核心执行循环。  
每个 Tick 必须可解释、可中断、可回放。

## 2. Tick 主循环
```c
while (!stable && tick < tick_limit) {
    gather_phase(scene);
    speculate_phase(scene);
    resolve_phase(scene);
    commit_phase(scene);
    tick++;
}
```

## 3. 第一步：Gather（意图收集）
输入：当前场景状态、规则表、计划 DAG。  
输出：本 Tick 的候选意图列表。

### 3.1 收集来源
1. 物理规则（重力、接触约束、流体扩散）。
2. 计划节点（例如抓取、移动、确认）。
3. 情绪门控（例如 Fear 提高回避倾向）。
4. 外部中断（高优先级系统事件）。

### 3.2 结果格式
每条意图包含：
1. `actor_instance_id`
2. `opcode`
3. `target`
4. `priority`
5. `expected_cost`

## 4. 第二步：Speculate（位移试探）
输入：候选意图。  
输出：可执行候选与冲突候选。

### 4.1 试探方法
1. 焦点区：位图位移 + 按位与检测。
2. 外围区：AABB 或质心距离近似。
3. 混合区：若外围命中风险阈值，升级为焦点精度。

### 4.2 试探目标
1. 判断动作是否可达。
2. 估算冲突代价。
3. 提前筛除必然失败动作。

## 5. 第三步：Resolve（冲突决断）
输入：可执行候选、冲突候选。  
输出：最终执行列表。

### 5.1 决断优先级
1. 安全规则优先。
2. 高优先级中断优先。
3. 高确定性计划优先。
4. 低代价动作优先（在同级情况下）。

### 5.2 冲突策略
1. 同实体互斥动作：择一执行。
2. 双向竞争资源：按策略仲裁或排队。
3. 情绪冲突：由 cognitive gate 限制可执行集。

## 6. 第四步：Commit（提交）
输入：最终执行列表。  
输出：更新后的场景状态与增量日志。

### 6.1 提交内容
1. 占位图更新。
2. 实例姿态与属性更新。
3. 关系边与计划节点状态更新。
4. 情绪寄存器衰减/增强。

### 6.2 提交保障
1. 提交必须原子化（逻辑原子，非硬件事务）。
2. 若任一步失败，回滚到 Tick 起点快照。

## 7. 稳定判定与终止条件
终止条件：
1. 达到稳定态。
2. 达到 `tick_limit`。
3. 遇到不可恢复错误。
4. 被外部中断终止。

## 8. 快进模式（Fast-forward）
用于长时程单调过程（如掉落、扩散）：
1. 识别单调段。
2. 合并多个 Tick 为批处理。
3. 在可证明安全的前提下跳过中间帧。

## 9. Tick 预算控制
每 Tick 设定操作预算：
1. `max_ops_focus`
2. `max_ops_periphery`
3. `max_conflict_resolve`
超预算则延迟低优先级意图到下一 Tick。

## 10. 伪代码：完整一次 Tick
```c
void run_tick(Scene32* s) {
    IntentQueue q = gather(s);
    CandidateSet c = speculate(s, q);
    ExecSet e = resolve(s, c);
    apply_exec(s, e);
    update_stability(s);
    trace_tick(s, q, c, e);
}
```

## 11. 常见失败模式
1. 意图震荡：同一对象来回执行互斥动作。
2. 冲突饥饿：低优先级动作长期无法执行。
3. 误稳定：外围低频导致局部变化未及时感知。

## 12. 修复策略
1. 引入最小执行窗口避免频繁翻转。
2. 对饥饿动作加 aging 提升优先级。
3. 定期做全局轻量巡检避免误稳定。

## 13. 指标
1. Tick 平均耗时
2. Tick p95 / p99
3. 冲突率
4. 回滚率
5. 稳定收敛步数

## 14. 本章结论
四步 Tick 流水线是 WVM 的执行心脏。  
其关键价值在于：每一步都可测、可控、可解释。
