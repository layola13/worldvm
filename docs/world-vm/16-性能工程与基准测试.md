# 16 性能工程与基准测试

## 1. 目标
建立可重复、可比较的性能评估体系，验证 WVM 的 CPU 友好目标。

## 2. 测试维度
1. 延迟：单次事件端到端时延。
2. 吞吐：单位时间可处理事件数。
3. CPU 占用：平均值与峰值。
4. 内存占用：常驻与峰值。
5. 收敛：达到稳定态所需 Tick。

## 3. 基准场景集
1. 轻任务：问答确认、点头反馈。
2. 中任务：抓取-移动-放置。
3. 重任务：多对象碰撞 + 计划中断恢复。
4. 心智任务：影子沙盒多候选评估。
5. 沟通任务：语言 + 手势联合输入。

## 4. 指标定义
1. `E2E_latency_ms`
2. `tick_avg_us`
3. `tick_p99_us`
4. `cpu_avg_pct`
5. `cpu_peak_pct`
6. `sandbox_alloc_per_sec`
7. `rollback_rate`
8. `replan_rate`

## 5. 基准目标（建议）
1. 轻任务 p95 < 20ms
2. 中任务 p95 < 80ms
3. 空闲 CPU < 1%
4. 高负载峰值可控且无长时间卡顿

## 6. Profiling 建议
1. Tick 各阶段耗时分解。
2. LOD 升降级开销分解。
3. 解析与调度开销分解。
4. 记忆检索开销分解。

## 7. 热点优化路径
1. 减少不必要的位图展开。
2. 优化跳表分发与分支预测。
3. 缓存热点规则与模板。
4. 合并可快进的 Tick 段。

## 8. 并行策略与约束
1. 并行优先用于“互不冲突 lane”。
2. 提交阶段保持确定性顺序。
3. 避免过度并行导致同步开销反噬。

## 9. 内存优化策略
1. 对象池复用 Scene 与临时结构。
2. 规则表和模板表连续存储。
3. 减少短生命周期堆分配。

## 10. 稳定性测试
1. 长时运行（24h）无内存泄漏。
2. 高频事件突发下无崩溃。
3. 持续中断恢复下无状态损坏。

## 11. 基准报告模板
每次测试输出：
1. 环境信息（CPU、内存、版本）
2. 场景说明
3. 指标结果（avg/p95/p99）
4. 回归对比
5. 优化建议

## 12. 自动化建议
1. CI 每日跑轻量基准。
2. 每周跑完整基准矩阵。
3. 版本发布前跑稳定性长测。

## 13. 本章结论
性能工程不是“最后再做”的工作。  
WVM 的核心卖点之一就是低占用，必须从设计阶段持续验证。
