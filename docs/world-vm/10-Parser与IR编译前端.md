# 10 Parser 与 IR 编译前端

## 1. 目标
将自然语言/动作输入稳定映射到可执行 IR，避免直接依赖概率生成作为执行依据。

## 2. 前端流程
1. 归一化（文本清洗、同义词归并）。
2. 词法分析（Lexing）。
3. 句法/语义解析（Parsing）。
4. 槽位抽取（实体、动作、条件、时间）。
5. IR 生成（Plan DAG + opcode 序列）。
6. 约束补全（默认参数、缺省对象推断）。

## 3. 词法层
### 3.1 词表结构
1. 高频词典（ROM 内）。
2. 领域扩展词典（插件）。
3. 同义映射表（canonical form）。

### 3.2 哈希查找
1. 使用整数词 ID，减少字符串比较。
2. 未登录词走回退策略：子词拆分或外部词典查询。

## 4. 语义槽位
最低要求提取：
1. `intent`：意图（问/做/确认/拒绝等）
2. `actor`：主动方
3. `target`：目标方
4. `action`：动作
5. `constraints`：约束（时间、地点、顺序、否定）

## 5. IR 结构建议
```c
typedef struct {
    uint16_t plan_id;
    uint16_t node_count;
    PlanNode nodes[MAX_NODES];
} PlanIR;
```

`PlanNode` 需包含：
1. `opcode`
2. `actor/target`
3. `predecessors`
4. `priority`
5. `timeout`

## 6. 默认推断规则
当输入不完整时：
1. 从上下文最近对象推断 target。
2. 从任务模板推断缺失步骤。
3. 若歧义过高，返回澄清问题，不盲猜执行。

## 7. 语义到指令映射
示例：
1. “请点头表示同意” -> `FOCUS(self) + NOD + CONFIRM`
2. “把杯子递给我” -> `FOCUS(cup) + GRAB(cup) + APPROACH(user) + RELEASE(cup)`
3. “别靠近它” -> `FOCUS(target) + MOVE_AWAY(target) + FREEZE(optional)`

## 8. 错误处理
1. 词法错误：未知词、非法符号。
2. 句法错误：结构不完整。
3. 语义错误：动作对象不匹配。
4. 执行错误：映射后无可执行模板。

每类错误都应返回：
1. 错误码
2. 位置信息
3. 可选修复建议

## 9. 可解释输出
前端需产出“解析证据”：
1. 命中词条
2. 触发模板
3. 推断项与置信度
4. 最终 IR

## 10. 与 LLM 的协作边界
1. 可用 LLM 辅助生成词典候选与模板候选。
2. 但执行前必须通过规则编译链验证。
3. 未通过验证的结果不能直接执行。

## 11. 语料扩展策略
1. 只采纳可映射为 IR 的样本。
2. 无法结构化的内容进入“观察池”待人工审阅。
3. 周期性生成“最小可执行模板”而非自由文本。

## 12. Parser 性能目标
1. 单句解析 p95 < 2ms（本地目标）。
2. 词法命中率 > 95%（日常域）。
3. 模板匹配回退率 < 10%。

## 13. 本章结论
Parser/IR 前端决定了系统是否“可执行”。  
只要输入能稳定编译成 IR，后续执行层就能保持确定性和可审计性。
