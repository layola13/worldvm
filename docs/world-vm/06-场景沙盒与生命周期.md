# 06 场景沙盒与生命周期

## 1. 目标
定义 `32x32x32` 场景沙盒（Scene32）的创建、执行、提交、销毁全过程。  
重点：沙盒是临时计算域，不是全局世界本体。

## 2. 核心原则
1. 无全局绝对坐标依赖：只处理当前事件局部相对关系。
2. 生命周期短：事件结束即释放，降低常驻成本。
3. 提交严格：只有稳定结果可写回全局关系图。

## 3. 场景结构建议
```c
typedef struct {
    uint64_t tick;
    uint32_t scene_id;
    uint32_t parent_event_id;

    uint64_t occupancy[512];      // 32^3 bit = 32768 bit = 4096B
    uint16_t instance_ids[128];   // 运行时实例槽
    uint8_t  instance_count;

    uint8_t  focus_x, focus_y, focus_z;
    uint8_t  focus_radius;

    uint8_t  state_flags;
    uint8_t  reserved[27];
} Scene32;
```

## 4. 生命周期阶段
1. `ALLOCATE`：从沙盒池申请空闲 Scene。
2. `INIT`：加载参与实体实例，设置焦点与初始规则。
3. `RUN`：按 Tick 推进直到稳定或达到步数上限。
4. `COMMIT`：将结果变更写回全局关系图。
5. `TEARDOWN`：清理内存并归还池。

## 5. 实例注入策略
1. 焦点对象必须全精度注入（16³ 拓扑）。
2. 关联对象按优先级注入，超限则降级为 AABB/质心。
3. 无关对象不注入，仅保留关系引用。

## 6. 坐标与锚点策略
1. 默认以“被动方/目标方”为局部锚点。
2. 主动方根据算子定义生成相对入场向量。
3. 介词（上/下/左/右/内/外）可修正初始相对位姿。

## 7. 稳定态判定
满足以下条件即稳定：
1. 本 Tick 没有有效位移意图。
2. 没有未处理冲突。
3. 没有高优先级中断。
4. 关键寄存器变化低于阈值。

## 8. 提交模型
提交不是“复制整个沙盒”，而是写变更集（Delta）：
1. 关系变化（新增/删除/权重变化）。
2. 实体属性变化（硬度、颜色、状态标记）。
3. 情绪寄存器变化（若启用心智层）。
4. 计划状态变化（节点完成/失败）。

## 9. 回滚机制
1. 提交前保存快照索引。
2. 如果提交冲突或策略守卫拒绝，执行回滚。
3. 回滚只恢复受影响节点，不全局重置。

## 10. 多沙盒并发
1. 多事件可并发运行多个 Scene。
2. 同一实体同时被多个 Scene 访问时，提交阶段做版本检查。
3. 冲突策略：高优先级事件优先，低优先级重算。

## 11. 错误恢复
1. 沙盒超时：强制停止并标记 `TIMEOUT`。
2. 无法稳定：触发降级计划或请求澄清。
3. 数据损坏：丢弃场景并上报诊断信息。

## 12. 资源回收
1. 采用对象池复用 Scene 结构。
2. `TEARDOWN` 阶段必须清零关键区块，防止脏状态泄漏。
3. 统计每个 Scene 生命周期耗时和峰值负载。

## 13. 可观察性
每个 Scene 应输出：
1. `scene_id`, `event_id`, `tick_count`
2. 参与实例数与 LOD 分布
3. 冲突次数、提交大小、回滚次数

## 14. 本章结论
通过短生命周期局部沙盒，WVM 将复杂推理转化为小规模可控仿真，确保性能和可解释性同时成立。
