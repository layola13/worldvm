# 17 MVP 范围与里程碑

## 1. 目标
定义最小可行产品（MVP）边界，确保项目可按阶段落地，不陷入无限扩展。

## 2. MVP 必须能力
1. 解析日常命令到 IR。
2. 执行基础动作计划（抓取、移动、确认、拒绝）。
3. 运行 32³ 沙盒与四步 Tick。
4. 启用焦点/外围 LOD 机制。
5. 提供基础情绪门控（至少 Fear/Calm）。
6. 输出可解释执行链。

## 3. MVP 不做
1. 大规模开放域知识问答。
2. 高复杂连续视觉任务。
3. 多语言全覆盖。
4. 跨设备分布式协同。

## 4. 里程碑规划
### M1：核心骨架（2~3 周）
1. Entity/Scene 基础结构
2. Tick 流水线
3. 最小 opcode 执行器

### M2：前端与计划（2~3 周）
1. Parser + IR
2. Plan DAG
3. Lane 调度器

### M3：LOD 与心智（2~3 周）
1. 焦点模糊机制
2. 影子沙盒
3. 情绪寄存器与门控

### M4：知识库与验收（2~3 周）
1. 10MB ROM 编译器
2. 场景回归测试
3. 性能与解释性验收

## 5. 交付物清单
1. 可运行 CLI Demo
2. ROM 样例包
3. 基准测试结果
4. 可解释日志样例
5. 开发者文档（本套）

## 6. 退出标准
满足以下全部条件即 MVP 完成：
1. 基础场景通过率 >= 95%
2. 可解释覆盖率 >= 99%
3. 关键性能目标达标
4. 内存预算不超限

## 7. 延期触发条件
1. 核心数据结构仍不稳定
2. Tick 回滚率持续过高
3. LOD 误判率不可接受
4. 解释链缺失严重

## 8. 版本节奏建议
1. 每周一个内部里程碑
2. 每两周一次可运行演示
3. 每阶段结束冻结接口

## 9. 风险缓冲
建议预留 20% 工期给：
1. 性能修复
2. 规则冲突修复
3. 文档与测试补齐

## 10. 本章结论
MVP 的核心是“可运行 + 可解释 + 可测量”。  
先把闭环跑通，再扩充知识与行为规模。
