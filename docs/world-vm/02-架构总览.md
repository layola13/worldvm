# 02 架构总览

## 1. 总体架构图（逻辑层）
WVM 由七层组成，从输入到输出依次为：
1. 输入层：文本、动作、上下文事件。
2. 前端层：词法解析、语义映射、IR 生成。
3. 指令层：Bagua ISA（6-bit opcode）与参数打包。
4. 运行时层：沙盒、Tick、调度器、碰撞与关系更新。
5. 心智层：影子沙盒、情绪寄存器、策略门控。
6. 知识层：10MB ROM + 外挂长期记忆。
7. 输出层：语言输出、动作输出、解释链输出。

## 2. 数据流主路径
1. 用户输入一句话或一个目标。
2. Parser 输出结构化语义节点（实体、关系、动作）。
3. 编译器将语义节点编译为指令序列与计划 DAG。
4. 调度器将指令投递到事件沙盒执行。
5. Tick 引擎推进状态，触发碰撞、路由、变更提交。
6. 结果写回关系图和记忆图谱。
7. 输出生成器给出执行结果与可解释日志。

## 3. 控制流主路径
### 3.1 正常路径
1. `PLAN_READY`
2. `EXECUTE_TICK`
3. `STABLE_STATE`
4. `COMMIT`
5. `RESPOND`

### 3.2 异常路径
1. `PLAN_CONFLICT`
2. `REPLAN`
3. `INTERRUPT`（安全中断、威胁中断）
4. `ROLLBACK`（回滚到稳定快照）

## 4. 模块划分
1. `wvm-parser`：解析器与词法哈希。
2. `wvm-ir`：语义 IR、opcode 编码、计划模板绑定。
3. `wvm-core`：实体、场景、Tick 主循环、提交器。
4. `wvm-lod`：视觉聚焦模糊、睡眠节拍、唤醒中断。
5. `wvm-mind`：影子沙盒、情绪场、认知门控。
6. `wvm-kb`：10MB 知识库编译与加载。
7. `wvm-store`：关系图、长期记忆、外部检索接口。
8. `wvm-observe`：日志、可视化、可解释链导出。

## 5. 运行模式
### 5.1 事件驱动模式（默认）
1. 系统空闲时几乎不计算，仅维护轻量状态衰减。
2. 收到事件后临时创建沙盒，完成推演后立即释放。
3. 极大降低平均 CPU 占用。

### 5.2 常驻仿真模式（可选）
1. 用于实验、训练、回放。
2. 保持 Tick 持续推进，便于观测长期演化。
3. 仅在调试/研究场景开启。

## 6. 状态一致性策略
1. 沙盒内状态是临时态，必须经过 `COMMIT` 才会写回全局。
2. 全局关系图是权威状态源（Source of Truth）。
3. 每次提交必须带版本号，支持冲突检测与幂等重放。

## 7. 错误处理策略
1. 解析错误：返回结构化错误码，不进入执行层。
2. 指令错误：标记非法 opcode，触发降级规则。
3. 计划冲突：执行重排，不可解则回退到安全响应。
4. 沙盒异常：销毁沙盒、回滚快照、记录审计日志。

## 8. 与传统 LLM 系统的关系
WVM 不是与 LLM 对立，而是定义主执行路径：
1. 主路径：规则 + 状态机 + 沙盒仿真（可解释）。
2. 辅路径：LLM 作为离线“资产生产器”或“建议器”。
3. 决策必须回到 WVM 可验证规则内执行。

## 9. 工程组织建议
1. 内核与知识分仓：避免耦合。
2. 数据格式先行：先定二进制布局，再写执行器。
3. 测试前置：每个模块提供最小可复现场景。
4. 指标导向：每个版本都必须看 CPU、延迟、解释率。

## 10. 本章结论
WVM 架构的核心是“事件驱动 + 局部仿真 + 可提交状态”。  
只要保持这条主线，系统就能在低算力条件下实现可控且可扩展的智能行为。
