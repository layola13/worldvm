## 架构补充说明（2026-04-24）

### 1. 一个有代表性的验收样例

后续物理内核是否真正成熟，不能只看“能不能下落”，而应看是否能自然承接如下场景：

- 铁球、玻璃球、保龄球
- 从高空跌落
- 分别撞到水泥地、泥地、水池

理想结果应能稳定区分：

- 反弹高度
- 碎裂与否
- 下陷深度
- 滚动与停止行为
- 入水冲击
- 浮沉与阻力变化

这个例子很重要，因为它能同时检验：

- 刚体积分
- 接触检测
- 材质配对
- 介质响应
- 破坏系统
- 观测/断言接口

### 2. 理想架构下，这类场景应如何被承接

一颗球从高空落下，理想的底层处理链路应为：

1. 刚体积分
2. 接触检测
3. 材质对材质响应计算
4. 地表/介质类型分流
5. 破坏与结构失效判定
6. 观测与 headless 断言输出

也就是说，真正成熟的内核，必须能表达：

- 什么物体
- 以什么速度
- 撞到什么介质
- 发生什么类型的响应

而不只是“是否被阻挡”。

### 3. 当前实现的真实边界

当前工程已经具备的，是以下基础：

- 原型级质量、摩擦、恢复系数、硬度、材质字段
- 基础重力与阻尼
- 简化回弹与摩擦衰减
- 基础破坏阈值判断
- 简化液体/浮力入口

但当前仍然缺失以下关键层：

- 材质对材质的统一配对规则
- 地表/介质类型闭环
- 统一查询层
- 稳定接触求解
- 泥地等软介质模型
- 水体入射与水下阻力模型
- 物理化的碎裂与应力传播

因此，当前系统更接近：

- 能表达“东西掉下去”

但还不能稳定表达：

- “什么东西，以什么方式，撞到什么介质后，会发生什么响应”

### 4. 下一阶段必须建立的底层分层

为了让上述样例可自然承接，底层需要明确分为以下模块：

#### A. Contact Material System

职责：

- 定义 `material A x material B -> response`

至少需要输出：

- restitution combine
- friction combine
- damage multiplier
- penetration resistance
- sink factor
- splash factor

说明：

- 不能再只依赖“单物体自身材质字段”
- 物理响应必须转向“材质对材质”

#### B. Surface & Medium System

职责：

- 统一描述接触对象属于哪种介质

至少应能区分：

- hard rigid surface
- soft deformable ground
- shallow water
- deep water

说明：

- 水泥、泥地、水池不能都被视为同一种“占用体素”
- 命中后必须进入不同响应路径

#### C. Unified Query Layer

职责：

- 为刚体、KCC、载具、人体提供统一的：
- voxel query
- overlap
- raycast
- sweep/cast
- penetration query

说明：

- 没有统一 Query，就无法保证球、水面、泥地、角色脚底、轮胎接地共享同一查询语义

#### D. Rigid Body Core

职责：

- 负责球体下落、滚动、接触、稳定 resting

说明：

- 这是所有材质与介质响应的基础承载层

#### E. Damage & Fracture System

职责：

- 负责玻璃球等脆性物体在冲击下的破坏

说明：

- 不能只依赖单阈值 `impact >= hardness`
- 后续应逐步引入接触条件、应力传播、碎片生成

#### F. Observation / Test API

职责：

- 为无头测试输出可断言的观测量

至少包括：

- 最大反弹高度
- 是否发生碎裂
- 下陷深度
- 入水事件
- 最终 resting 状态
- 最终速度

### 5. 当前优先级上的含义

这类“材质球体落到不同介质”的场景说明了一件事：

- 当前最缺的不是继续增加零散物理接口
- 而是补齐统一的运行时骨架与查询层

因此，下一阶段的优先方向应保持为：

1. 统一 `PhysicsWorld`
2. 统一 `Query Layer`
3. 接触与材料配对系统
4. 介质类型系统
5. 破坏与观测接口升级
