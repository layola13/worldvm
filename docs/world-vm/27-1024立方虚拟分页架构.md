# 01 1024立方虚拟分页架构

## 1. 背景
`todo2.md` 的核心扩展是：把世界寻址从 `32³` 升级到 `1024³`。  
关键结论：`1024³` 不能作为“整块物理数组”常驻，必须采用虚拟分页。

## 2. 规模与约束
1. 原始位图规模：`1024³ / 8 = 128MB`（仅占位，不含元数据）。
2. 系统预算：总内存仍按 `100MB` 微内核设计。
3. 因此：`1024³` 只能是逻辑空间，物理内存只保留活跃页。

## 3. 虚拟分页模型
1. 逻辑空间：`0..1023` 三轴整数坐标。
2. 宏分页：将全空间划分为 `32 x 32 x 32` 宏块（每块可映射到 32³ 或 16³ 局部容器）。
3. 活跃页池：仅为“当前事件相关块”分配实际内存。
4. 空白块：不分配实体数据，只保留空洞标记。

## 4. 分层访问策略
1. 第一级：页表查找（是否驻留、是否活跃）。
2. 第二级：粗粒度包围体判断（AABB）。
3. 第三级：必要时才加载细粒度位图或公式采样。

## 5. 事件驱动执行
1. 输入触发事件后，才激活相关分页。
2. Tick 只遍历活跃分页，不扫描全空间。
3. 稳定态后释放非必要页，回写关系与状态。

## 6. 优势
1. 维持 1024³ 寻址能力，同时控制实际内存占用。
2. 支持城市级/多场景逻辑推演，而不牺牲 CPU 可运行性。
3. 与现有 16³ 单体资产和 32³ 局部沙盒兼容。

## 7. 风险与控制
1. 风险：频繁页切换造成抖动。  
控制：引入热页缓存和滞回阈值。
2. 风险：大范围事件触发页爆炸。  
控制：设置每 Tick 页预算和优先级降级策略。

## 8. v1 执行建议
1. 先实现“逻辑页 + 活跃池 + 简单双层查找”。
2. 先不做复杂八叉树压缩，避免过早复杂化。
3. 优先把页生命周期和稳定态回收跑通。
