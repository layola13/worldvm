# 11 计划系统与并行调度

## 1. 目标
将“想做什么”稳定编译成“可执行步骤”，并在低成本下支持确定性并行。  
核心输出是 `Plan DAG + Lane Scheduler`。

## 2. 为什么不是单线程脚本
单线程顺序脚本有三个问题：
1. 无法利用自然并行（如左右手同时动作）。
2. 关键路径变长，响应变慢。
3. 一处阻塞会拖垮全计划。

## 3. Plan DAG 结构
每个计划是有向无环图：
1. 节点：原子动作（opcode + 参数）。
2. 边：依赖关系（必须先完成）。
3. 标签：优先级、失败策略、超时。

建议结构：
```c
typedef struct {
    uint16_t node_id;
    uint8_t  opcode;
    uint16_t actor;
    uint16_t target;
    uint16_t preds[8];
    uint8_t  pred_count;
    uint8_t  lane_hint;
    uint8_t  priority;
    uint8_t  timeout_tick;
    uint8_t  retry_limit;
    uint8_t  status; // PENDING/RUNNING/DONE/FAILED
} PlanNode;
```

## 4. Lane 设计
首版建议 5 个 lane：
1. `LEFT_HAND`
2. `RIGHT_HAND`
3. `SPEECH`
4. `GAZE`
5. `PLANNER`

每 Tick 同一 lane 最多执行一个节点，避免同资源冲突。

## 5. 调度流程
1. 找到所有前置满足的 `ready` 节点。
2. 根据 lane 与优先级分桶。
3. 每个 lane 选择一个最优节点。
4. 运行冲突检测，必要时重排。
5. 提交执行，更新节点状态。

## 6. 优先级规则
1. 安全相关节点优先。
2. 关键路径节点优先（减少总时长）。
3. 低重试节点优先（避免级联失败）。
4. 同级按最早进入时间优先。

## 7. 失败与重试
节点失败后策略：
1. 可重试：指数退避重试，直到 `retry_limit`。
2. 可替代：切换备选模板。
3. 不可恢复：触发计划级失败并回滚。

## 8. 计划中断
中断来源：
1. 威胁事件（如碰撞风险暴增）。
2. 用户取消。
3. 资源不足。

中断处理：
1. 标记当前计划 `PAUSED`。
2. 将中断计划插入高优先级队列。
3. 中断结束后恢复原计划。

## 9. 计划恢复
恢复时需满足：
1. 关键上下文仍有效。
2. 依赖对象状态未失效。
3. 超时窗口未过。

否则进入 `REPLAN` 重新编译。

## 10. 计划模板与参数化
模板定义为“骨架 + 槽位”：
1. 骨架固定步骤。
2. 槽位绑定实体 ID、位置、数量。
3. 约束绑定时间/先后/否定条件。

## 11. 示例：做三明治（简化）
1. `FOCUS(bread)`  
2. `LEFT_HAND: GRAB(bread)`  
3. `RIGHT_HAND: GRAB(knife)`  
4. `RIGHT_HAND: SPREAD(sauce)`（依赖 2,3）  
5. `BIND(layer)`  
6. `CONFIRM(done)`

步骤 2 和 3 可并行。

## 12. 调度伪代码
```c
void schedule_plan(Plan* p, Scene32* s) {
    ReadySet rs = collect_ready_nodes(p);
    LanePick lp = pick_per_lane(rs);
    ExecSet ex = resolve_lane_conflicts(lp, s);
    run_exec(ex, s);
    update_plan_status(p, ex);
}
```

## 13. 常见问题
1. 死锁：互相等待。  
解决：拓扑检查 + 超时强制破环。
2. 饥饿：低优先级节点长期不执行。  
解决：aging 增权。
3. 抖动：节点频繁暂停恢复。  
解决：最小运行窗口。

## 14. 指标与验收
1. 计划完成率
2. 平均完成 Tick 数
3. 并行利用率
4. 中断恢复成功率
5. 重规划率

## 15. 本章结论
“DAG + 多 lane 确定性并行”是 WVM 计划执行的默认模型。  
它在保持可解释性的同时，显著降低顺序执行带来的性能损耗。
