# 26 外挂 LLM 与百科字典层规范

## 1. 目标
定义 WVM 如何接入外部大模型（LLM）作为“字典 + 百科 + 候选解释器”。  
关键约束：外挂 LLM 是**知识协处理器**，不是执行主内核。

## 2. 设计原则
1. 主执行路径仍由 WVM 规则引擎控制（Parser -> IR -> Plan -> Tick）。
2. 外挂 LLM 只提供“候选知识项”和“结构化解释草稿”。
3. 任何外部返回都必须经过本地验证、映射和审计，才可进入执行层。
4. 高风险场景默认拒绝“仅凭外部 LLM”直接决策。
5. 物理相关判断优先使用本地 headless 工具检查，LLM 只参与解释和候选生成。

## 3. 架构位置
外挂层位于 `L3 External Store`：
1. `WVM Core`：确定性执行与状态提交。
2. `Memory Graph`：本地关系与缓存。
3. `LLM Adapter`：向外部模型发起查询，返回结构化候选。
4. `Verifier`：本地规则校验、来源打分、冲突处理。

## 4. 职责边界
### 4.1 外挂 LLM 负责
1. 词义补充（术语解释、同义词扩展）。
2. 百科事实候选（人物、地理、历史、概念）。
3. 非核心领域描述性内容补全。
4. 将长文本归纳为候选三元组/关系边。

### 4.2 外挂 LLM 不负责
1. 直接生成可执行 opcode 并落地运行。
2. 覆盖本地安全守卫与策略规则。
3. 在低置信情况下跳过澄清流程。
4. 直接写入高置信长期记忆。

## 5. 查询触发条件
仅在以下情况触发外部查询：
1. 本地词典未命中（OOV）。
2. 本地知识置信不足且任务需要外部事实。
3. 用户明确要求百科解释。
4. 计划编译缺少关键语义槽位。

不触发情况：
1. 本地规则已足够完成任务。
2. 高实时任务（延迟预算不足）。
3. 安全敏感且外部来源不可验证。

## 6. 查询协议（建议）
外部请求使用结构化包：
```json
{
  "query_id": "uuid",
  "intent": "ASK_INFO|DEFINE|DISAMBIGUATE",
  "language": "zh-CN",
  "topic": "string",
  "constraints": {
    "max_tokens": 600,
    "require_sources": true,
    "time_bound": "2026-04-23"
  },
  "expected_schema": "fact_candidates_v1"
}
```

## 7. 外部返回格式（必须结构化）
```json
{
  "query_id": "uuid",
  "candidates": [
    {
      "subject": "实体A",
      "relation": "属于/位于/提出于",
      "object": "实体B",
      "confidence_ext": 0.72,
      "source_hint": "外部模型说明或引用"
    }
  ],
  "summary": "可选摘要",
  "uncertainty_notes": "不确定项"
}
```

## 8. 本地验证流水线
外挂结果进入执行前，必须经过：
1. `Schema Check`：字段完整性校验。
2. `Rule Check`：关系类型合法性校验。
3. `Conflict Check`：与本地高置信记忆冲突检测。
4. `Source Check`：来源分级与新鲜度评估。
5. `Trust Score`：计算本地综合置信。

仅通过阈值的候选可进入“可用知识池”。

## 9. 置信分级策略
定义本地信任等级：
1. `T0` 未验证（仅临时参考，不落库）。
2. `T1` 结构合法（可用于澄清问题）。
3. `T2` 与本地关系一致（可用于低风险回答）。
4. `T3` 多源一致（可写入长期记忆候选区）。
5. `T4` 人工/权威确认（可作为高置信事实）。

## 10. 冲突处理
当外挂结果与本地知识冲突：
1. 不覆盖本地高置信事实。
2. 记录冲突边到 `conflict_bucket`。
3. 输出时明确“不一致，需确认”。
4. 如任务关键，触发用户澄清或人工审阅。

## 11. 缓存与成本控制
1. 查询结果写入短期缓存（TTL）。
2. 相同 query 使用语义键去重。
3. 高成本请求启用预算阈值（每会话最大外部请求数）。
4. 低价值重复问题优先用本地缓存回复。

## 12. 延迟控制
1. 外挂查询需设置超时（如 800ms~1500ms）。
2. 超时则回退本地保守回答。
3. 在计划执行链路中仅允许异步外挂，不阻塞关键动作。

## 13. 安全与隐私
1. 对外请求前做脱敏（用户标识、私密字段）。
2. 敏感任务默认禁用外部查询。
3. 外部返回内容不直接执行、不直接持久化。
4. 记录审计日志：谁查了、查了什么、用了什么结果。

## 14. 与日常语言映射联动
与第 25 章配合方式：
1. 第 25 章先做本地意图与槽位抽取。
2. 若槽位缺失，再触发外挂补全候选。
3. 候选回流第 25 章映射器，重新编译 CommIR。
4. 若仍低置信，发起 `CLARIFY`。

## 15. 与 3D 引擎联动
外挂知识只能影响“语义层”，不能绕过几何/物理事实：
1. 可补“对象定义”“常识关系”。
2. 不可跳过碰撞/重力/约束判定。
3. 最终行为仍由 Scene32 + Tick 引擎决定。

## 16. 与 headless 物理/化学检查联动（补充）
当请求属于高风险物理或具身语义（运动、碰撞、容器、摄入）：
1. LLM 先输出结构化候选动作与假设参数。
2. Tool Call 触发本地 headless 检查（物理/化学/规则）。
3. Verifier 合并结果，生成 `PASS/FAIL + repair_hint`。
4. 仅 `PASS` 候选可进入执行层。

## 17. 典型流程示例
输入：`“帮我解释一下量子隧穿是什么意思，并举个日常类比。”`

流程：
1. 本地识别 `ASK_INFO + DEFINE(topic=量子隧穿)`。
2. 本地词典无完整定义，触发外挂 LLM 查询。
3. 返回候选定义与类比草稿。
4. 本地验证结构与冲突。
5. 输出“解释文本”，但不生成执行动作。

## 18. 评测指标
1. 外挂命中收益率（相比纯本地）。
2. 低置信误用率（应尽量低）。
3. 冲突检出率。
4. 外挂平均延迟与超时率。
5. 单会话查询成本。
6. 触发工具检查后的误判率与漏判率。

## 19. 本章结论
外挂 LLM 应该是 WVM 的“百科外脑”，不是“执行中枢”。  
只要保持“外部候选 + 本地验证 + 主内核执行”三段式，系统就能同时获得知识广度与执行确定性。
