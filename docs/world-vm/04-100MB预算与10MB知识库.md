# 04 100MB 预算与 10MB 知识库

## 1. 目标
本章定义内存预算、分配原则、监控指标和超限处理策略。  
重点：知识库 `ROM` 首版上限固定为 `10MB`，不得突破。

## 2. 全局预算（100MB）
建议初版分配如下：

| 模块 | 预算 | 说明 |
|---|---:|---|
| 核心运行时代码与常量 | 10MB | VM 内核、调度器、基础规则 |
| ROM 知识库（硬上限） | 10MB | 词汇、模板、微码、规则 |
| 沙盒池与临时缓冲 | 8MB | 多沙盒并发 + 调试缓存 |
| 关系图与短期记忆 | 12MB | 全局关系与活动上下文 |
| 长期记忆索引缓存 | 10MB | 外挂记忆的本地高速索引 |
| 观测与日志环形缓冲 | 5MB | Trace、统计、解释链 |
| 系统预留 | 45MB | 插件、快照、回滚与扩展 |

## 3. 10MB 知识库细分
建议按用途切分：

| 子模块 | 预算 | 内容 |
|---|---:|---|
| 词汇与语义映射 | 3.0MB | 高频词表、同义映射、槽位定义 |
| 计划模板库 | 2.0MB | 日常任务 DAG 模板 |
| 动作/手语微码 | 1.5MB | 点头、摇头、抓取、挥手等 |
| 情绪触发表 | 1.0MB | 情绪寄存器门控与响应规则 |
| 常识关系种子图 | 1.5MB | 人-物-动作基础关系 |
| 版本与预留区 | 1.0MB | 升级头、兼容占位 |

## 4. 数据格式策略
1. 采用二进制定长块，避免 JSON/XML 冗余。
2. 所有 ID 使用整数索引，减少字符串重复。
3. 高频结构使用位字段压缩。
4. 规则优先索引引用，避免长文本重复存储。

## 5. 知识项粒度原则
1. 存“可执行模板”而非“冗长描述文本”。
2. 存“关系结构”而非“百科段落”。
3. 存“操作符 + 参数”而非“逐帧动画”。
4. 复用基础动作，组合成复杂行为。

## 6. 预算监控机制
1. 构建期：编译器输出每个区块字节统计。
2. 启动期：加载器校验总大小与段校验和。
3. 运行期：周期采样驻留内存与峰值。
4. CI：若 ROM 超过 10MB，构建失败。

## 7. 超限处理策略
1. 第一步：去重词条与模板。
2. 第二步：压缩低频规则，延迟加载。
3. 第三步：迁移到外挂记忆，保留索引。
4. 第四步：按业务优先级裁剪。

## 8. 版本兼容策略
1. ROM 头部写入 `magic/version/segment_count/checksum`。
2. 升级只允许“向后兼容字段追加”。
3. 关键枚举值固定，不随版本重排。
4. 提供迁移工具：`rom-migrate vA -> vB`。

## 9. 示例：ROM Header（建议）
```c
typedef struct {
    uint32_t magic;          // 'WROM'
    uint16_t major;          // 主版本
    uint16_t minor;          // 次版本
    uint32_t total_size;     // 总字节数
    uint32_t segment_count;  // 段数量
    uint32_t crc32;          // 完整性校验
} WvmRomHeader;
```

## 10. 性能与预算联动
预算不是只看“能否装下”，还要看“读取成本”：
1. 高频段放连续内存，提高缓存命中。
2. 稀有段可分离到冷区，按需映射。
3. 热路径配置尽量控制在 L2/L3 可承受范围。

## 11. 指标建议
1. ROM 加载时间 < 200ms（目标机器）。
2. 热路径规则查询 p99 < 50us。
3. 词汇映射命中率 > 95%（日常域）。
4. 计划模板复用率 > 80%。

## 12. 本章结论
10MB 知识库不是限制，而是架构约束。  
它强迫系统采用“高复用、低冗余、可执行模板优先”的正确方向。
