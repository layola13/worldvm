# 40 时间因果逻辑层

## 1. 目标
补齐 WVM 在常识时序推理上的规则能力，处理“存在性、先后顺序、因果一致性、角色生效窗口”。

## 2. 设计定位
1. 时间因果层是独立规则裁判，不替代物理/化学/人宇宙。
2. 职责是判定“是否成立”，不是执行“如何发生”。
3. 默认位于 Parser/IR 与执行层之间。

## 3. 典型问题
1. “父母结婚时为什么不请儿子”
2. “先毕业再入学”
3. “结果先于原因”
4. “实体未出生却参与事件”

## 4. 最小数据结构（文档级）
```c
typedef struct {
    int entity_id;
    int64_t born_at;
    int64_t active_from;
    int64_t active_to;
} EntityTimeline;

typedef struct {
    int event_id;
    int64_t start_at;
    int64_t end_at;
} EventSpan;

typedef struct {
    int from_event;
    int to_event;
    int relation; // BEFORE / AFTER / DURING / CAUSES
} TemporalEdge;
```

## 5. 核心规则集（v1）
1. 存在性规则：实体参与事件时必须 `exists(entity, event_time)`。
2. 顺序规则：`BEFORE(A,B)` 要求 `A.end <= B.start`。
3. 因果规则：`CAUSES(A,B)` 要求 `A.start <= B.start`。
4. 角色规则：角色未生效时禁止执行对应动作。
5. 区间合法性：所有事件满足 `start <= end`。
6. 冲突处理：规则冲突时返回 `FAIL + reason_code + break_time`。

## 6. 与执行链的关系
1. `Parser -> IR` 后先跑 `Temporal Check`。
2. `PASS`：进入物理/化学/人宇宙执行。
3. `FAIL`：触发澄清或修复提示，不进入高风险执行链。

## 7. 返回协议建议
```json
{
  "status": "PASS|FAIL",
  "reason_code": "T_EXISTENCE_VIOLATION|T_ORDER_CONFLICT|T_CAUSAL_CONFLICT",
  "break_time": 123456789,
  "repair_hint": "..."
}
```

## 8. 示例：父母结婚与儿子邀请
1. `marriage.start = t_marriage`
2. `son.born_at = t_birth`
3. 若 `t_birth > t_marriage`，则 `exists(son, t_marriage) = false`
4. 结论：邀请条件不成立，不属于“故意不请”。

## 9. 测试建议
1. 正常样例：先出生后入学、先结婚后生子。
2. 冲突样例：未出生参与、结果早于原因。
3. 回归要求：同输入同输出 reason_code 与 break_time。

## 10. 本章结论
时间因果逻辑层是 WVM 的“常识一致性守门员”。  
它与物理/化学裁判并列，保障系统在叙事和关系推理中的基础正确性。
