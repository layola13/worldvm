# 18 测试矩阵与验收标准

## 1. 目标
提供完整测试框架，覆盖功能正确性、性能、稳定性、可解释性与安全性。

## 2. 测试分层
1. 单元测试：结构体、编码解码、单指令语义。
2. 组件测试：Parser、Scheduler、LOD、Mind 模块。
3. 集成测试：输入到执行到输出全链路。
4. 回归测试：版本升级后的行为一致性。
5. 压力测试：并发事件和长时运行。

## 3. 功能测试矩阵（示例）
| 类别 | 用例 | 期望 |
|---|---|---|
| 解析 | “把杯子拿过来” | 生成有效 IR |
| 调度 | 双手并行动作 | 无冲突执行 |
| 碰撞 | 锤子砸玻璃 | 触发破碎变更 |
| LOD | 外围静态场景 | 低频更新且不漏关键事件 |
| 心智 | 高威胁输入 | 触发回避门控 |
| 沟通 | 点头/摇头 | 对应确认/拒绝语义 |

## 4. 性能验收
1. 定义目标机器配置。
2. 跑固定场景集至少三轮取中位数。
3. 必须输出 avg/p95/p99。
4. 不达标必须附优化计划。

## 5. 可解释性验收
每条关键执行链需具备：
1. 输入证据
2. 指令序列
3. 状态变化
4. 结果与原因码

抽检通过率目标：`>= 99%`。

## 6. 安全验收
1. 禁止规则必须可命中拦截。
2. 对高风险动作必须二次确认或拒绝。
3. 回滚路径可正常工作。

## 7. 稳定性验收
1. 24 小时无崩溃、无泄漏。
2. 高频中断下无状态损坏。
3. 长链计划无死锁。

## 8. 兼容性验收
1. ROM 跨版本加载成功。
2. 老模板在新版本语义一致。
3. 未实现 opcode 有合理降级。

## 9. 自动化建议
1. 每次提交跑单元与关键组件测试。
2. 每日跑集成回归。
3. 每周跑压力与长稳。

## 10. 缺陷分级
1. P0：崩溃、数据损坏、越权执行。
2. P1：核心功能错误、严重性能退化。
3. P2：局部逻辑偏差、解释链缺失。
4. P3：非关键可用性问题。

## 11. 验收报告模板
1. 版本信息
2. 覆盖率统计
3. 失败用例清单
4. 性能指标
5. 风险清单与处置

## 12. 本章结论
没有测试矩阵的“智能系统”不可维护。  
WVM 的每个版本都应以可验证结果作为上线前提。
