# 35 精简物理引擎架构

## 1. 章节目标
本章补齐“物理引擎”层面的完整设计，和第 23 章“物理算子”形成上下层关系：
1. 第 23 章定义“算子规则”。
2. 本章定义“引擎架构、执行管线、模块边界、数据流和验收标准”。

## 2. 设计边界
### 2.1 要做
1. 离散体素世界中的刚体/流体/气体基础模拟。
2. 整数坐标 + 位运算 + 有限状态更新。
3. 与 `1024³` 虚拟分页协同的按需激活计算。

### 2.2 不做
1. 连续高精度刚体求解器（如全量约束迭代器）。
2. 高阶 Navier-Stokes 连续流体方程。
3. 面向渲染的材质与视觉特效系统。

## 3. 引擎分层
物理引擎建议拆成 5 层：
1. `PhysicsState`：状态存储层（实体、速度、材质、热态）。
2. `BroadPhase`：粗筛（AABB / 分页活跃区过滤）。
3. `NarrowPhase`：细判（位图/公式采样碰撞）。
4. `Dynamics`：动力学更新（重力、推力、扩散、反应）。
5. `Resolver`：冲突决断与结果提交（含回滚点）。

## 4. 数据模型
### 4.1 核心状态
每个实例维护：
1. `position_i32`（整数坐标）
2. `velocity_i16`（定点/离散速度）
3. `material_type`
4. `flags`（static/dynamic/breakable 等）
5. `thermal_state`

### 4.2 分页绑定
1. 实例记录 `page_id` 与 `local_offset`。
2. 物理计算只在活跃页上执行。
3. 跨页运动由页管理器负责迁移和重映射。

## 5. 执行管线（每 Tick）
1. **Collect**：收集受力与意图（重力、外力、脚本动作）。
2. **BroadPhase**：基于 AABB 与页索引筛掉不可能碰撞对。
3. **NarrowPhase**：对候选对做位图或 SDF 精判。
4. **Resolve**：判定阻挡、反应、破碎、扩散结果。
5. **Commit**：提交位置/状态变化，写入 Delta 与 Trace。

## 6. 三种物理类型
### 6.1 刚体（Rigid）
1. 以占位/碰撞为主，速度离散更新。
2. 默认规则：重力下移、障碍阻挡、冲击触发材料反应。

### 6.2 流体（Fluid）
1. 采用元胞自动机风格扩散。
2. 优先级：`DOWN > SIDE > DIAGONAL > HOLD`。
3. 只计算活跃前沿，避免全场遍历。

### 6.3 气体（Gas）
1. 低密度扩散模型，偏向上浮与均匀扩散。
2. 使用简化密度字段，不做连续压力场求解。

## 7. 与 1024³ 虚拟分页协同
1. 物理引擎消费页调度器提供的活跃页清单。
2. 每 Tick 设置页级计算预算，防止激活风暴。
3. 非活跃页不做细判，仅保留摘要状态。
4. 关键事件可强制唤醒邻近页（带滞回阈值）。

## 8. 决定性与可回放
1. 同输入 + 同随机种子必须得到同输出。
2. 每 Tick 输出 `trace`（碰撞对、规则 ID、变更摘要）。
3. 支持从任一检查点回放并比对状态一致性。

## 9. 引擎接口建议
```c
init_physics(world_state)
step_physics(tick, active_pages, intents)
query_collision(entity_a, entity_b)
apply_external_force(entity, vec3_i16)
dump_physics_trace(tick_range)
```

## 10. 性能与验收
### 10.1 功能验收
1. 苹果落桌稳定。
2. 锤子砸玻璃触发破碎。
3. 水体溢出与沿边扩散。
4. 气体在封闭空间扩散并受边界约束。

### 10.2 性能验收（建议）
1. 轻场景 Tick p95 达标（按项目基线）。
2. 活跃页数量增加时性能近线性增长。
3. 空闲场景接近零额外 CPU 占用。

## 12. 结论
WVM 的物理引擎是“离散逻辑求解器”，不是传统图形物理插件。  
它的价值是提供可解释、可约束、可回放的现实约束层，为上层语言与模型输出兜底。

## 13. 架构补强：材质配对 + 介质系统 + Query 层
为支撑复杂场景（如不同材质球体坠落到不同地表/介质），本章补充三项引擎级能力。

### 13.1 材质配对系统（Material Pair System）
从“单体材质属性”升级为“接触配对规则”：
1. 输入：`material_a`, `material_b`, `surface_condition`, `medium_type`, `relative_speed`
2. 输出：`restitution`, `friction`, `damage_factor`, `sink_factor`, `splash_factor`

约束：
1. 配对规则优先于默认组合规则。
2. 组合策略必须固定且可回放。
3. 任意平台结果一致。

### 13.2 地表/介质系统（Surface & Medium）
接触必须先分类介质再进入响应：
1. `hard_surface`（如水泥）
2. `soft_surface`（如泥地）
3. `liquid_shallow`（浅水）
4. `liquid_deep`（深水）

分流效果：
1. 硬地：冲量与摩擦主导。
2. 软地：下陷与能量耗散主导。
3. 水体：入水冲击、浮力、阻力与飞溅事件主导。

补充：
1. 同一 `material_type` 在不同 `surface_condition` 下可产生不同响应。
2. 例如 `CONCRETE + DRY`、`CONCRETE + WET`、`CONCRETE + ICY` 必须允许走不同接触分支。
3. “冰块”与“结冰地面”不能混为一谈，前者偏本体材质，后者偏表面状态。

### 13.3 统一查询层（Unified Query Layer）
查询层应作为 Solver 前置输入，不再由各模块自行拼装查询逻辑。

最小 API：
1. `queryVoxel`
2. `raycastSingle / raycastAll`
3. `overlapCapsule`
4. `capsuleCast`
5. `computePenetration`

过滤器统一支持：
1. `ignore_self`
2. `body_type_filter`
3. `sensor_policy`
4. `layer_mask`
5. `environment_policy`

查询结果除几何命中信息外，建议可选返回：
1. `material_type`
2. `surface_condition`
3. `medium_type`
4. `body_type`

这样 Resolver 不必再次从多个子系统拼接接触分类信息。

## 14. 执行管线升级（每 Tick）
在原有 5 步管线上，补充“介质分流与材质配对”。

建议升级为：
1. **Collect**：收集外力、控制输入与活跃对象。
2. **Query/BroadPhase**：统一查询入口生成候选接触。
3. **NarrowPhase**：位图/SDF 精判，计算命中法线与 TOI。
4. **Classify**：判定地表/介质类型。
5. **PairResolve**：解析材质配对参数。
6. **Dynamics/Resolve**：冲量、摩擦、下陷、浮力、破坏。
7. **Commit**：提交状态、事件、Delta、Trace。
8. **Snapshot**：生成可回放快照与哈希数据。

### 14.1 推荐数据流
建议在 `NarrowPhase -> Classify -> PairResolve -> Resolve` 之间固定传递以下结构：
1. `QueryHit`
2. `ContactClassification`
3. `MaterialPairResponse`
4. `ContactTelemetry`

说明：
1. `QueryHit`：几何命中结果。
2. `ContactClassification`：材质、表面状态、介质、body type。
3. `MaterialPairResponse`：供 Resolver 消费的接触参数。
4. `ContactTelemetry`：供 Trace、回放、headless 断言使用。

## 15. 接触响应路径（引擎视角）
建议固定“单接触处理路径”：
1. `detect_contact`
2. `classify_surface_medium`
3. `resolve_material_pair`
4. `solve_impulse_and_friction`
5. `evaluate_damage_or_break`
6. `apply_medium_post_effect`
7. `emit_trace_event`

### 15.1 ContactClassification（文档级最小模型）
建议定义：

```c
struct ContactClassification {
    int material_a;
    int material_b;
    int surface_condition;
    int medium_type;
    int body_type_a;
    int body_type_b;
}
```

用途：
1. 统一 Resolver 输入。
2. 避免 KCC、Vehicle、RigidBody 各自重算分类。
3. 便于 Trace 记录与回放比对。

### 15.2 MaterialPairResponse（文档级最小模型）
建议定义：

```c
struct MaterialPairResponse {
    float restitution;
    float friction_static;
    float friction_dynamic;
    float damage_factor;
    float sink_factor;
    float splash_factor;
    float drag_factor;
}
```

说明：
1. `drag_factor` 主要服务水体、泥浆、松散介质。
2. `sink_factor` 不只服务泥地，也可用于雪地、沙地等软介质。

### 15.3 事件建议
1. `CONTACT_BEGIN`
2. `CONTACT_PERSIST`
3. `CONTACT_END`
4. `BREAK`
5. `SPLASH`
6. `SINK_UPDATE`

## 16. 观测接口规范（Engine Telemetry）
为确保可解释与无头验收，建议统一输出：
1. `impact_speed`
2. `max_bounce_height`
3. `sink_depth`
4. `break_event_count`
5. `splash_intensity`
6. `rest_state`
7. `material_pair_id`
8. `surface_medium_type`
9. `query_cost_us`
10. `solver_cost_us`

说明：
1. 这些指标应进入 Trace 与回放系统。
2. 必须可用于跨版本回归对比。

### 16.1 ContactTelemetry（文档级最小模型）
建议定义：

```c
struct ContactTelemetry {
    float impact_speed;
    float bounce_height;
    float sink_depth;
    float splash_intensity;
    int break_event_count;
    int rest_state;
    int material_pair_id;
    int surface_condition;
    int medium_type;
}
```

用途：
1. 服务无头断言。
2. 服务版本间回归比较。
3. 服务网络回放与 determinism 检查。

## 17. 新验收样例（补充）
新增一组跨系统验收样例：
1. 铁球/玻璃球/保龄球从同一高度落到水泥地。
2. 同三种球落到泥地，验证下陷和能量损耗差异。
3. 同三种球入水，验证入水冲击和浮沉行为差异。
4. 相同输入重复两次，验证事件序列与观测值一致。

### 17.1 验收字段
每组样例至少断言：
1. 首次撞击速度
2. 第一次反弹高度
3. 是否触发破坏
4. 最终稳定状态
5. 若为软地/水体则断言下陷或飞溅字段

### 17.2 扩展材质与表面状态样例
建议继续补以下文档级验收：
1. 铁球落到 `CONCRETE + DRY`
2. 铁球落到 `CONCRETE + ICY`
3. 玻璃球落到 `MUD + WET`
4. 保龄球落到 `ASPHALT + WET`
5. 橡胶球落到 `ICE + DRY`
6. 布包落到 `WATER + DEEP`

每组样例至少比较：
1. `surface_condition` 是否影响摩擦/回弹
2. `medium_type` 是否影响后处理路径
3. `material_type` 是否影响破坏概率和损伤等级

### 17.3 人体/载具关联样例
同一套分类与响应模型后续还应服务：
1. 鞋底 `RUBBER` 在 `CONCRETE + DRY/WET/ICY` 上的差异
2. 轮胎 `RUBBER` 在 `ASPHALT/GRAVEL/MUD/SNOW` 上的差异
3. 骨骼 `BONE` 与 `CONCRETE/ASPHALT` 的受击差异
4. 布料 `COTTON_CLOTH` 入水后的阻力与下垂差异

## 19. 本章补充结论
精简物理引擎并不意味着“仅保留重力和阻挡”。  
真正可扩展的精简架构应具备：
1. 统一查询入口
2. 材质配对机制
3. 介质分流路径
4. 可回放的观测接口
5. 可扩展的材质/表面状态体系

这样才能让 KCC、载具、人体和网络回滚共享同一底层语义。
