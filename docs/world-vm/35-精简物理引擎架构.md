# 35 精简物理引擎架构

## 1. 章节目标
本章补齐“物理引擎”层面的完整设计，和第 23 章“物理算子”形成上下层关系：
1. 第 23 章定义“算子规则”。
2. 本章定义“引擎架构、执行管线、模块边界、数据流和验收标准”。

## 2. 设计边界
### 2.1 要做
1. 离散体素世界中的刚体/流体/气体基础模拟。
2. 整数坐标 + 位运算 + 有限状态更新。
3. 与 `1024³` 虚拟分页协同的按需激活计算。

### 2.2 不做
1. 连续高精度刚体求解器（如全量约束迭代器）。
2. 高阶 Navier-Stokes 连续流体方程。
3. 面向渲染的材质与视觉特效系统。

## 3. 引擎分层
物理引擎建议拆成 5 层：
1. `PhysicsState`：状态存储层（实体、速度、材质、热态）。
2. `BroadPhase`：粗筛（AABB / 分页活跃区过滤）。
3. `NarrowPhase`：细判（位图/公式采样碰撞）。
4. `Dynamics`：动力学更新（重力、推力、扩散、反应）。
5. `Resolver`：冲突决断与结果提交（含回滚点）。

## 4. 数据模型
### 4.1 核心状态
每个实例维护：
1. `position_i32`（整数坐标）
2. `velocity_i16`（定点/离散速度）
3. `material_type`
4. `flags`（static/dynamic/breakable 等）
5. `thermal_state`

### 4.2 分页绑定
1. 实例记录 `page_id` 与 `local_offset`。
2. 物理计算只在活跃页上执行。
3. 跨页运动由页管理器负责迁移和重映射。

## 5. 执行管线（每 Tick）
1. **Collect**：收集受力与意图（重力、外力、脚本动作）。
2. **BroadPhase**：基于 AABB 与页索引筛掉不可能碰撞对。
3. **NarrowPhase**：对候选对做位图或 SDF 精判。
4. **Resolve**：判定阻挡、反应、破碎、扩散结果。
5. **Commit**：提交位置/状态变化，写入 Delta 与 Trace。

## 6. 三种物理类型
### 6.1 刚体（Rigid）
1. 以占位/碰撞为主，速度离散更新。
2. 默认规则：重力下移、障碍阻挡、冲击触发材料反应。

### 6.2 流体（Fluid）
1. 采用元胞自动机风格扩散。
2. 优先级：`DOWN > SIDE > DIAGONAL > HOLD`。
3. 只计算活跃前沿，避免全场遍历。

### 6.3 气体（Gas）
1. 低密度扩散模型，偏向上浮与均匀扩散。
2. 使用简化密度字段，不做连续压力场求解。

## 7. 与 1024³ 虚拟分页协同
1. 物理引擎消费页调度器提供的活跃页清单。
2. 每 Tick 设置页级计算预算，防止激活风暴。
3. 非活跃页不做细判，仅保留摘要状态。
4. 关键事件可强制唤醒邻近页（带滞回阈值）。

## 8. 决定性与可回放
1. 同输入 + 同随机种子必须得到同输出。
2. 每 Tick 输出 `trace`（碰撞对、规则 ID、变更摘要）。
3. 支持从任一检查点回放并比对状态一致性。

## 9. 引擎接口建议
```c
init_physics(world_state)
step_physics(tick, active_pages, intents)
query_collision(entity_a, entity_b)
apply_external_force(entity, vec3_i16)
dump_physics_trace(tick_range)
```

## 10. 性能与验收
### 10.1 功能验收
1. 苹果落桌稳定。
2. 锤子砸玻璃触发破碎。
3. 水体溢出与沿边扩散。
4. 气体在封闭空间扩散并受边界约束。

### 10.2 性能验收（建议）
1. 轻场景 Tick p95 达标（按项目基线）。
2. 活跃页数量增加时性能近线性增长。
3. 空闲场景接近零额外 CPU 占用。

## 11. 与现有章节关系
1. 第 22 章：几何表示与碰撞几何基础。
2. 第 23 章：具体算子规则表。
3. 第 27-29 章：1024³ 分页和地址体系。
4. 本章：将以上内容组装成可实现的“物理引擎”。

## 12. 结论
WVM 的物理引擎是“离散逻辑求解器”，不是传统图形物理插件。  
它的价值是提供可解释、可约束、可回放的现实约束层，为上层语言与模型输出兜底。
