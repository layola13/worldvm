# 02 三世界分页与总线同步

## 1. 目标
把 `todo2.md` 中“实体世界 + 心理世界 + 编程世界”落为可实现的分页协同方案。

## 2. 三世界定义
1. **物理世界页（Physical Page）**：拓扑、碰撞、重力、流体。
2. **心理世界页（Psychological Page）**：情绪寄存器、意图状态、语义权重。
3. **编程世界页（Programming Page）**：规则、算子、元数据与可修改配置。

## 3. 同步原则
1. 三页共享 `entity_id` 与 `instance_id` 作为跨页主键。
2. 不共享具体存储布局，只共享消息与状态摘要。
3. 任何跨页修改必须走总线（Bus）提交，禁止隐式写入。

## 4. 总线消息模型
建议消息结构：
1. `msg_type`（PHYSICS_EVENT / AFFECT_EVENT / RULE_UPDATE）
2. `source_page`
3. `target_page`
4. `entity_ref`
5. `payload`
6. `tick_id`
7. `priority`

## 5. 同步时序
1. Tick 内先跑物理页（硬约束）。
2. 物理结果产生事件后广播到心理页与编程页。
3. 心理页更新门控参数，再回写下一 Tick 的调度偏好。
4. 编程页仅在规则合法时发放配置更新。

## 6. 防页错误策略
1. 未驻留页收到消息时进入“延迟处理队列”。
2. 超时未加载则降级为摘要模式。
3. 对必须立即处理的高优先级消息触发强制唤醒。

## 7. 典型例子
1. 物理页检测“碰撞失败” -> 心理页 `arousal` 上升。
2. 心理页“恐惧升高” -> 调度器缩小焦点半径。
3. 编程页调整“避障阈值” -> 下一 Tick 物理页更新策略。

## 8. 边界控制
1. 物理页不可被心理页直接覆盖硬规则（例如重力仍然生效）。
2. 编程页变更必须通过策略守卫与版本校验。
3. 总线必须可追踪、可回放、可审计。

## 9. 实施优先级
1. v1 先做物理页主导 + 心理页只读响应。
2. v1.1 再开放编程页有限规则更新。
3. v2 再考虑三页并行细粒度协同。
