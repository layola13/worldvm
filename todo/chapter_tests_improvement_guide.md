# 章节测试改进指导

## 目标

这份说明是给当前项目测试编写者的指导意见。重点不是批评你写了很多文件，而是帮助你把“看起来很多测试”改造成“真正能证明功能已经实现的测试”。

当前 `src/chapter*.zig` 里的大部分章节测试，更接近“场景占坑”或“章节目录映射”，还不能算严格意义上的验收测试。

你接下来要做的，不是继续批量加章节文件，而是把每一章都改造成：

1. 明确测试目标
2. 明确依赖的底层模块
3. 明确输入
4. 明确可验证输出
5. 明确失败条件

---

## 对当前测试的总体评语

### 优点

1. 你已经把很多章节主题组织出来了，说明你知道系统最终想覆盖哪些能力。
2. 场景命名大体有方向感，例如 `ccd_bullet`、`hinge_door`、`kcc_slope_walk`，能看出你想按物理主题分层。
3. `runChapterTest` 的结构统一，后续重构时可以复用。

### 主要问题

1. 测试只有“运行了几帧”的断言，没有验证物理结果。
2. 章节名和真实执行逻辑经常不一致。
3. 测试没有绑定对应模块，很多只是把几个实例丢进通用 tick。
4. 测试通过条件过弱，几乎无法发现错误实现。
5. 测试名称写得很强，但实际验证很弱，容易制造“已经完成”的错觉。

---

## 你现在最需要建立的测试观念

### 观念一：测试不是摆场景，是验证因果关系

错误写法：

- 放一个“子弹”
- 放一堵“墙”
- 跑 100 tick
- 断言 `ticks_to_stable > 0`

这不能证明 CCD 工作了。

正确思路：

- 给子弹一个高速速度
- 记录起点、终点、碰撞时刻
- 断言它没有穿墙
- 断言碰撞发生在墙前
- 断言 TOI 在 `0..1` 范围内

### 观念二：测试名必须配得上测试内容

如果测试名字叫：

- `motor_joint`
- `cloth_draping`
- `gravity_zone_high`

那么测试里就必须真的：

- 创建 motor joint
- 调用软体或布料求解逻辑
- 对局部重力场施加影响

否则就应该重命名成：

- `placeholder_motor_joint_scene`
- `softbody_future_scene`
- `force_zone_stub_scene`

### 观念三：章节测试必须绑定章节对应模块

例如：

- KCC 章节必须调用 `kcc` 模块，不允许只拿苹果自由落体代替角色控制器。
- Joints 章节必须显式创建 joint，并调用关节求解。
- CCD 章节必须调用 CCD/TOI 相关逻辑，不能只依赖普通离散步进。
- 力场章节必须调用 `applyExplosion`、`applyForceField` 或等价机制。

---

## 如何判断一个章节测试是不是合格

一个合格测试，至少要回答下面五个问题：

1. 我在验证哪个具体功能？
2. 我调用了哪个真实模块？
3. 输入是什么？
4. 期望输出是什么？
5. 如果模块坏了，这个测试会不会红？

如果第五个问题回答不出来，这个测试大概率是无效测试。

## 对当前各类章节测试的具体指导

### 一类：可以保留，但必须加强断言

这类通常是基础场景还算合理，只是断言太弱。

适合保留的方向：

1. 球体碰撞
2. 堆叠
3. 斜坡
4. 水平滑动
5. 自由落体

你要做的改进：

1. 记录初始位置、最终位置、速度变化
2. 对比关键坐标是否符合预期
3. 检查状态是否从 `idle/falling` 变成 `resting`
4. 检查是否发生反弹、减速、停止

示例：

- “球从 20 高度落下后，最终 `pos_y` 不应继续下降到地板以下”
- “有阻尼时，10 tick 后 `abs(vel_x)` 应小于初值”
- “无摩擦时，水平速度变化应小于阈值”

### 二类：必须重写，因为测试名和内容脱节

典型对象：

1. `chapter11_kcc`
2. `chapter19_joints`
3. `chapter20_ccd`
4. `chapter23_softbody`
5. `chapter25_forces`

这些章节现在的问题不是“断言弱一点”，而是“根本没测到该测的模块”。

你的处理方式应该是：

1. 先确认底层模块是否真的存在可调用入口
2. 如果存在，改测试去直接调用模块
3. 如果不存在，暂时不要伪装成完成测试
4. 把文件降级为 `placeholder` 或移动到场景目录

### 三类：当前应改名为占坑场景

如果底层模块根本没接入主流水线，就不要叫 test。

建议改名方式：

- `chapter20_ccd.zig` 中未接入 CCD 的场景，改为 `ccd_scenes_placeholder.zig`
- `chapter23_softbody.zig` 改为 `softbody_scenes_placeholder.zig`
- `chapter25_forces.zig` 改为 `forcefield_scenes_placeholder.zig`

这样做不是退步，而是诚实。

## 推荐的重构方法

### 第一步：把“场景生成”和“断言逻辑”分离

现在每个章节文件都在重复：

1. setup
2. 跑 tick
3. 收集结果
4. 简单断言

建议提炼成通用工具：

- `spawn chapter scenario`
- `run fixed ticks`
- `capture positions/velocities/states`
- `assert stable / assert moved / assert collided / assert bounced`

你应该把可复用逻辑收敛到一个公共 helper 里，而不是每章复制一份 `runChapterTest`。

### 第二步：每个测试只验证一个核心性质

错误方式：

- 一个测试同时想验证关节、碰撞、摩擦、睡眠、破坏

正确方式：

- 一个测试只验证“固定关节是否保持距离”
- 一个测试只验证“铰链是否限制某轴旋转”
- 一个测试只验证“高速物体是否穿过薄墙”

### 第三步：断言要尽量数值化

避免这种话：

- “应该看起来像摆动”
- “应该比较稳定”
- “应该像是没穿透”

改成数值断言：

- 两物体距离误差小于 1 voxel
- 最终 `pos_y >= floor_y`
- 反弹后 `vel_y > 0`
- 10 tick 后速度衰减至少 20%
- 命中时刻 `toi < 1.0`

### 第四步：失败条件要能对应具体 bug

好的测试失败后，开发者能立刻知道问题在哪。

例如：

- “对象穿透地板”通常对应碰撞检测或 depenetration 问题
- “关节距离漂移过大”对应约束求解问题
- “高速物体跳过薄墙”对应 CCD 问题
- “角色在斜坡上不受控滑落”对应 KCC 或摩擦问题

## 各章节的具体作业

### 作业 A：重写 KCC 章节

当前问题：

- 只是把普通实体当角色代理
- 没有调用 KCC 创建、更新、碰撞、滑墙逻辑

你要完成：

1. 使用真实 KCC API 创建角色
2. 给角色输入移动、跳跃、蹲伏
3. 分别验证平地、斜坡、台阶、矮洞
4. 断言角色高度、位移、是否接地、是否被阻挡

验收标准：

1. 平地前进有位移
2. 斜坡移动不会穿地
3. 台阶可爬或不可爬表现一致
4. 蹲伏前后角色高度有差异

### 作业 B：重写 Joints 章节

当前问题：

- 文件名叫 joints，但没有真正创建 joint 参与 step

你要完成：

1. 显式创建 fixed / hinge / slider / spring joint
2. 在 step 前后测量相对距离或相对角度
3. 对 breakable joint 设置断裂阈值
4. 如果 motor 未实现，不要假装测试已完成

验收标准：

1. fixed joint 两端距离变化在阈值内
2. hinge 不允许非法轴向偏移
3. slider 沿非允许轴的偏移应接近 0
4. spring 在多 tick 下出现振荡或回归趋势

### 作业 C：重写 CCD 章节

当前问题：

- 名字很高级，但没有验证真正 CCD 行为

你要完成：

1. 直接调用 TOI/CCD 相关函数
2. 构造薄墙、高速物体、小目标
3. 比较“不开 CCD”与“开 CCD”的差异
4. 检查是否发生穿透

验收标准：

1. 高速物体在普通离散检测下可能穿透
2. 开 CCD 后应停在障碍前
3. TOI 必须在合法范围内
4. 不能只断言“跑过了”

### 作业 D：清理 Softbody 章节

当前问题：

- 当前内核没有真正软体章节所需的求解基础

你要完成：

1. 先判断底层软体能力是否存在
2. 若不存在，把本章改成 `placeholder`
3. 补一份“未来需要的底层清单”

验收标准：

1. 不再用普通刚体伪装软体测试
2. 文件名与真实状态一致

### 作业 E：重写 Force Field 章节

当前问题：

- 力场章节没有施加任何真实力场

你要完成：

1. 调用 `applyExplosion` 或 `applyForceField`
2. 为风场、爆炸、浮力分别设计不同断言
3. 记录施力前后速度和位移变化

验收标准：

1. 爆炸后对象径向远离中心
2. 浮力后对象垂直速度上升
3. 风场后对象沿风方向位移增加

## 建议的测试文件结构

建议你把测试分成三层：

### 第一层：模块单测

例如：

- `joint.zig` 的约束求解函数
- `ccd.zig` 的 TOI 计算
- `kcc.zig` 的滑墙和高度切换

特点：

- 输入直接
- 输出明确
- 定位 bug 快

### 第二层：系统集成测试

例如：

- TickEngine + Joints
- TickEngine + Terrain + Material Pairing
- Query + CCD + Collision

特点：

- 验证模块接线是否正确

### 第三层：章节验收测试

例如：

- “Chapter 20: CCD thin wall should not tunnel”
- “Chapter 19: fixed joint should preserve anchor distance”

特点：

- 面向功能里程碑
- 不要太多，但要有代表性

你现在最大的问题，是把第三层写成了第一层和第二层都不算的中间态。

---

## 编写断言时的最低标准

以后每写一个新测试，至少满足下面四条：

1. 至少一个和物理意义相关的断言
2. 至少一个失败后可定位问题的错误信息
3. 至少一个和章节主题直接相关的模块调用
4. 测试名和实际逻辑一致

如果做不到，就不要叫“章节测试”。

---

## 分阶段执行计划

### 第一周

1. 删除或降级所有明显名实不符的章节测试
2. 抽取公共 `runScenario` / `captureState` helper
3. 给基础运动、碰撞、摩擦补真实断言

### 第二周

1. 重写 KCC 章节
2. 重写 Joints 章节
3. 重写 CCD 章节

### 第三周

1. 把未落地的 Softbody / ForceField 章节改为 placeholder
2. 补充 TODO 和缺失底层说明
3. 建立“模块单测 -> 系统集成 -> 章节验收”的分层规范

## 最后给学生的话

你现在的问题不是“不努力”，而是“太快进入了命名和铺量阶段”，导致测试文件数量很多，但信息密度很低。

真正好的测试，不在于文件多，而在于：

1. 它能约束实现
2. 它能在实现错误时可靠失败
3. 它能告诉别人这项功能到底有没有真的完成

所以接下来要收缩，不要继续扩写假的章节测试，而是把少量关键章节先做扎实。

先把 5 个测试写成真的，再谈 50 个测试。
