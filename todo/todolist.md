## 执行顺序说明

这份清单保留原始分层与编号，便于追踪，但**实际开发顺序不应机械按章节线性执行**。

当前建议以本节为准：

1. 先完成 `Layer 1 / Phase 1`
   - 统一几何、查询协议、QueryHit 语义、layer/group/filter
   - 固定 `tick_engine` 权威执行语义与 `physics_world` orchestrator 语义
2. 再完成 `Layer 1 / Phase 2`
   - 接触分类、PairResolve、Impulse/Friction、Sleep/Medium 主链
   - 这部分是 KCC / Vehicle / Ragdoll / Ballistics 的共同底座
3. 再完成 `Layer 1 / Phase 3`
   - Constraint / Joint / CCD / Crash Defense / Force Field
   - 注意：`physics_kernel` 的 joint runtime/protocol 收口已经明显先于本清单描述，不应按“从零开始”的心态重做
4. 将 `Layer 1 / Phase 4` 中的 `Rewind + Determinism` 前移
   - `trace / snapshot / hash / rewind / determinism`
   - 这部分应在大规模推进 Layer 2 前成为硬底座
5. `Layer 1.5` 只先做预测核心与消费者统一
   - 先推进 `251-320`
   - `321-400` 的大量验证场景和高级预测特性后置
6. 再进入 `Layer 2`
   - 推荐顺序：`KCC -> Vehicle/Tire/Suspension/Drivetrain -> Ballistics/Destruction -> Ragdoll -> AI Traffic/Sensors/Planner/Safety -> Disaster/Terrain/Weather`
7. `Layer 3` 全部后置
   - 研究域不应阻塞统一物理底座落地
8. `工程与测试基础设施` 不应最后集中做
   - 测试、回归、性能基准、可视化、诊断要伴随每层同步推进

## 并行边界

下面这些块可以并行，但必须遵守前置依赖：

1. 可并行
   - QueryHit 协议扩展 与 query 性能优化
   - Contact classify 与 telemetry/trace 输出
   - KCC 与 Ballistics/Destruction
   - Vehicle 与 AI Traffic/Sensors 的接口对齐
2. 不建议并行
   - Query 语义未锁定前推进 KCC/Vehicle 主逻辑
   - Contact/Constraint 未稳定前推进 Ragdoll/高级 Vehicle 动力学
   - Snapshot/Determinism 未成型前推进网络回滚或大规模预测消费者
3. 必须后置
   - Layer 3 研究域
   - 大规模高级预测特性
   - 纯形式化或高阶测试基础设施
