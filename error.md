# WVM 设计评估报告

## 一、总体判断

这是一份**目标清晰、约束严格、工程化意识强**的设计规范。核心思路是"用离散拓扑 + 事件驱动 + 可执行规则替代稠密矩阵运算"，在 CPU 友好和可解释性上有真实追求，而非空想。设计上最大的特点是**约束即架构**——10MB ROM 上限、4KB 实体结构、6-bit ISA 这三个硬约束驱动了所有上层决策，这是正确的工程思路。

但也必须指出：文档的完备程度远超实现进度，部分设计细节存在循环依赖或悬空定义，MVP 边界不够清晰，某些模块（如影子沙盒、外挂 LLM）的设计深度不足。

---

## 二、优势分析

### 1. 约束驱动架构，方向正确

10MB ROM 硬上限和 100MB 总预算不是营销话术，而是真正的设计约束。它迫使系统采用"可执行模板优先、高复用、低冗余"的正确方向——这与当前主流 LLM 路线形成了鲜明对比。如果真正落地，在嵌入式或低功耗场景中是有竞争力的。

### 2. Tick 流水线设计扎实

四步（`Gather → Speculate → Resolve → Commit`）的定义清晰，每步都有明确的输入输出、失败模式和回滚策略。快进模式（单调段合并）和 Tick 预算控制的设计也体现了对性能的真实关注。特别是"每步可测、可控、可解释"这一原则，贯穿整个设计。

### 3. ISA 设计简洁实用

6-bit 定长指令、Opcode 分段（控制/空间/物理/认知）、每条指令定义前置条件/读写集/错误码/可回滚性——这些是工业级虚拟机的好实践。Jump Table 路由层不做业务判断也是一个好的分离。

### 4. 实体 ABI 稳定性有保障

4KB 固定大小、512B 拓扑位图、分区明确的字段布局、版本演进规则（只追加不改变既有字段）——这为多语言实现（文档提到 Rust/C/Zig）和长期迭代打下了基础。

### 5. 心智层设计有创意

四维情绪寄存器（valence/arousal/certainty/control）配合影子沙盒，为"情绪如何影响决策"提供了可执行的机制而非模糊描述。门控规则和情绪原型（Fear/Love/Sadness）的具体化也有利于验收测试的编写。

## 三、问题与风险

### 1. 固定 16³ 体素拓扑是最大的架构风险

文档说这是"离散世界推理"的核心，但 16³=4096 个体素对于任何稍微复杂的场景都极其受限。如果要表达一个有十来个物体的厨房，可能就捉襟见肘了。文档中 22-24 章涉及的 3D 引擎、体素空间、物理算子都与这个基础强相关，一旦扩展到 1024³ 虚拟分页（27-28 章），整个拓扑编码逻辑都要重写。这个基础决策值得重新审视。

### 2. 影子沙盒定义悬空

这是心智层最关键的设计，但它只定义了概念和工作流，缺少以下关键细节：
- 影子沙盒与真实沙盒的状态同步协议是什么？
- 影子沙盒内部的 Tick 精度是否与主沙盒一致？
- "低精度近似"具体是什么级别的近似？位图精度？物理精度？
- 如果影子沙盒的预测结果与实际执行不符，如何处理？

没有这些，影子沙盒就变成了一个"先跑一遍试试"的模糊概念。

### 3. 情绪规则系统的膨胀风险

四维寄存器本身是简洁的，但情绪更新规则和门控规则目前只有示例（文档中不超过 10 条），而实际系统可能需要数十甚至上百条规则来覆盖各种场景。文档提到的风险（规则过多难以调试、寄存器震荡）没有给出具体的技术缓解方案，只说了"规则分层、按域启用"和"平滑与滞回"这样的方向性建议，缺乏可执行性。

### 4. Parser 与知识库的边界模糊

文档提到日常语言映射（25 章）和外挂 LLM（26、32 章），但没有说清楚：当用户输入一句模糊的自然语言时，是走 ROM 模板匹配还是调用外挂 LLM？两者的分工、切换条件和置信度协议完全缺失。这个边界如果没划清楚，实现时会变成大量的临时补丁。

### 5. 可解释性的实现代价被低估

"输入 → 指令 → 状态转移 → 结果"的解释链在文档中多次被强调，但这意味着每一层都要维护完整的 trace。Tick 级 trace、指令级 trace、情绪驱动 trace 加在一起，日志量会非常大。文档在第 4 章给观测缓冲只留了 5MB，在第 7 章又要求 trace every tick——这两件事在 5MB 预算下根本无法同时成立。

### 6. MVP 范围与文档体量严重不匹配

文档有 36 章，涵盖了 3D 引擎、物理算子、虚拟分页、外挂 LLM 数据工厂等大量内容，但文档自己也承认 v1.x MVP 只做"日常沟通与计划执行示例"。这个体量的规范对于一个 MVP 来说完全是过度工程，会导致项目在"规范完成"之前根本无法交付可运行的东西。建议 MVP 相关的工作全部聚焦在 01、02、04、05、07、09、12 这七章之内。

## 四、具体设计疑点

| 位置 | 疑点 |
|---|---|
| 05 章 拓扑编码 | bit_index 映射公式 `z + x*16 + y*16*16` 与标准体素索引不一致（通常 z 是最内层），需要验证 |
| 07 章 Speculate | "AABB 或质心距离近似"在外围区如何保证与焦点区的一致性？ |
| 09 章 ISA | opcode 编号"尽量不改"但没有版本化的 opcode 描述语言，未来兼容性存疑 |
| 12 章 门控 | `arousal > 220` 触发避险、`certainty < 40` 触发 ASK——这些阈值是固定值还是可学习的参数？ |
| 04 章 预算 | 系统预留 45MB（几乎一半）但没有说明用途，这笔钱可能反复被挪用 |
| 06 章 场景沙盒 | "32x32x32 事件沙盒"与"16x16x16 固定体素拓扑"的关系没有说明——是沙盒内可以容纳多个实体，还是实体本身是 16³ 的？ |

## 五、总结

| 维度 | 评分 | 说明 |
|---|---|---|
| 愿景清晰度 | ★★★★★ | 目标明确，差异化路线清晰 |
| 架构一致性 | ★★★★☆ | 层次清晰，但部分模块边界模糊 |
| 工程可行性 | ★★★☆☆ | 约束严格是好事，但部分设计过于理想化 |
| 可解释性 | ★★★★★ | 这是该设计最核心的价值，完成度较高 |
| 实现风险 | ★★★☆☆ | MVP 范围不清，影子沙盒等核心设计有悬空 |
| 文档质量 | ★★★★☆ | 详尽但体量过大，与实现阶段不匹配 |

**核心建议**：现在最需要做的是**停止写文档、开始写代码**。选定一个 7 章的 MVP 子集（01/02/04/05/07/09/12），用 Zig 或 Rust 实现可运行的最小内核，跑通 "输入一句话 → Parser → ISA 指令 → Tick 执行 → 输出结果" 的完整路径。只有在真实代码上跑过，才能发现文档中那些看似合理实则悬空的设计。文档的作用应该是记录已经验证的决策，而不是预先规划所有可能性。
